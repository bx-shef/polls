# Индекс решений

> Короткое «почему так» по ключевым архитектурным решениям. Продуктовые решения
> (№1–8) — в [`brief.md`](./brief.md) §12; здесь — технические решения ядра.
> Принцип: каждое нетривиальное решение фиксируется (конвенция `CLAUDE.md`).

| Решение | Почему так |
|---|---|
| **Brain-first** | Продаём не анкеты, а решения: AI-слой превращает ответы (вкл. фритекст) в число (CSAT/NPS) + причину + действие. Опрос — самый дешёвый датчик, защищённый ров — AI-анализ. См. `brief.md` §13. |
| **`invitationPolicy` — version-frozen (#21)** | Политика приглашений вшита в схему/версию, а не в рантайм-конфиг: ответ должен интерпретироваться по той версии, на которой собран. Иммутабельность версии распространяется и на политику. |
| **Презентация (`intro`/`thanks`/`blocks`) — version-frozen (#25)** | Контент экранов Интро/Спасибо и имена блоков вшиты в `SurveyDraft`/`CompiledVersion` и едут в `compiled_schema` — единый источник правды для Vue-слоя (а не второй источник/хардкод), заморожен со снимком анкеты. `blocks` — упорядоченный массив имён (минимально), `question.block` остаётся именем; `SurveyFill` не трогает (он про прохождение). |
| **AES-256-GCM с `kid` в blob** | Шифрование `portal.tokens` с идентификатором ключа (`kid`) внутри blob → форвард-совместимость ротации ключа без миграции данных. `loadTokenKey` — startup-guard (падает на пустом/плейсхолдерном ключе). См. `bitrix24/crypto.ts`. |
| **Keyset-пагинация (не OFFSET)** | Курсор по стабильному ключу даёт стабильную выборку при вставках и постоянную стоимость страницы (OFFSET деградирует на хвосте). Helpers — `store/cursor.ts`. |
| **Подавление малых N в слое чтения** | Анонимность гарантируется там, где данные отдаются (PgStore/read-API), а не в «сырых» агрегатах — нельзя обойти, читая агрегат напрямую. |
| **Схема — единый источник истины (zod)** | Типы и runtime-валидация выводятся из одних zod-схем → TS-тип и проверка не расходятся. `domain/schema.ts`. |
| **Драйвер-агностичный PgStore** | `Queryable` ≈ `pg.Pool`/pglite: те же `.sql`-миграции применяют и прод-Postgres, и pglite-тесты (единый источник схемы, паритет с in-memory). |
| **Logger — интерфейсо-зависим, не вендоро** | Ядро знает интерфейс `Logger`, прод подменяет адаптером (Pino/Sentry) инъекцией — zero-dep ядро, без вендор-лока. См. `docs/observability.md`. |
| **Локальное приложение (1 портал = 1 инстанс)** | Решение №7: своя БД на сервере клиента, без кросс-портальной мультитенантности — проще изоляция и приватность; tenant-ключ `portalId`. |
| **Публичная проекция версии — в HTTP-хендлере, не в сторе/схеме** | `GET /api/survey/:key/current` отдаёт версию БЕЗ `invitationPolicy` (триггер-стадии/канал — внутренняя CRM-конфигурация). Отсечение в `toPublicVersion` (api-слой), а не в `IStore`/схеме: стор и аналитика политику видят, наружу не утекает только публичный read-контур. Тип `Omit<…,'invitationPolicy'>` гейтит будущие чувствительные поля компилятором. |
