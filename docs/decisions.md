| **Дашборд контура B — серверный агрегат, dev-only, гейт на чистом сторе (#34/#47)** | `/api/dashboard/:key` считает NPS/CSAT/распределение СЕРВЕРНО через `domain/aggregate` над общим `useStore` (те же ответы, что собирает submit). Вопросы — по `metric` из текущей версии (не хардкод seed-ключей). Малые N подавлены (`meetsAnonymity`) — первичный барьер анонимности. Auth-гейт — `requirePortalSession` (см. запись «Auth дашборда (#47)» ниже): fail-closed в проде, dev/гейт открыты; tenant-изоляция по `portalId` → #49. Визуальный гейт детерминирован за счёт инварианта «гейт-тесты НЕ пишут в стор» (submit в thanks/submit-error замокан) — иначе живой агрегат «плывёт» по числу ответов. Срезы: тренд NPS (`npsTrend`, подавление точек по месяцу) и срез по услугам (`byProduct`, имя из денормализованного `context.products[].productName`; метрику показываем только при её СОБСТВЕННОЙ выборке ≥ порога, а не только по числу ответов услуги). Срезы (услуга/направление/ответственный/клиент) — ядровой `breakdownBy` (под тестами; имена денормализованы в `CrmContext`: `companyName`/`dealCategoryName`/`responsibleName`/`productName` — снимок подписи на закрытии сделки, как `productName`; фолбэк на ID; денормализация ради читаемых срезов без обращения к CRM-справочникам, контекст в JSONB lossless → миграция не нужна; `responsibleName` — PII под #31). `breakdownBy` — in-memory над `ResponseRecord[]`; SQL-вариант для PgStore-пути дашборда — #49. Фильтр по версии (`?version=N`): состояние держим в URL (а не в `ref` компонента) — деплинкуемо, кнопка «назад» работает, и SSR-дружелюбно (гейт снимает срез без клика, `useAsyncData` рефетчит по смене query). Доступные версии считаем ДО фильтра (селектор не «схлопывается»); невалидную/чужую версию игнорируем (все версии). |
| **Auth дашборда (#47) — подписанная сессия портала, fail-closed в проде** | `requirePortalSession` (`server/utils/auth.ts`) гейтит `/api/dashboard/:key`: прод (`DASHBOARD_AUTH_SECRET`) требует валидную подписанную сессию (`src/api/session.ts` — HMAC-SHA256 токен `{portalId, exp}`, не шифрование: payload не секрет, но неподделываем; constant-time сверка). Сессию минтит handshake Bitrix24 app-фрейма (ядро `src/bitrix24/frame.ts` — сделано, см. запись «Handshake app-фрейма» ниже; привязка эндпоинта/cookie → #49). Почему НЕ NODE_ENV для dev-открытости: собранный Nitro-сервер бежит как `production` (гейт упёрся бы в fail-closed), поэтому открытость — по ЯВНОМУ `DASHBOARD_DEV_OPEN=1` (гейт/`pnpm preview` ставят флаг; локальный `pnpm dev` открыт по `NODE_ENV!=='production'`). Дефолт без секрета и без флага в проде → 503 (а не тихая отдача PII). Секрет короче `MIN_SECRET_LEN` (32) → 503 (не используем слабый HMAC); dev-open логирует громкий warn один раз. Решение гейта — чистая `resolveDashboardAuth` (без env/h3, под unit-тестами: все ветки 401/503/dev). Требование к будущему минту (handshake #47): cookie `polls_portal` — `HttpOnly` + `Secure` + `SameSite` (внутри iframe Bitrix24 — вероятно `None; Secure`, проверить на портале). tenant-фильтрация по `portalId` — на PgStore (#49); здесь только seam «кто спрашивает». |
| **Handshake app-фрейма Bitrix24 (#47) — ядро-рантайм, secure-by-design** | `src/bitrix24/frame.ts`: параметры `BX24.getAuth` (`DOMAIN`/`member_id`/`AUTH_ID`/…) приходят POST'ом на наш ПУБЛИЧНЫЙ эндпоинт → недоверенны. Два барьера: (1) `DOMAIN` управляем злоумышленником → SSRF при REST-вызове → `isAllowedPortalDomain` (allowlist `*.bitrix24.<tld>`, голый хост, без slash/port); (2) `member_id` НЕ берём из сырого POST (иначе со своим валидным токеном выпишешь сессию на чужой tenant) → `verifyFrameAuth` берёт member_id из АВТОРИТЕТНОГО `authenticate` (живая проверка токена, инжектируется) и сверяет с заявленным, расхождение → отказ. `mintPortalSession` подписывает сессию (`signSession`). Почему ядро-первым (без эндпоинта): боевой `authenticate` (#49) делает ЛЁГКИЙ REST-вызов с `AUTH_ID` (НЕ OAuth-refresh — он ротирует токен → race при параллельных загрузках фрейма), `AUTH_ID` — в теле/заголовке, не в query (утечка в логи); нужен `PortalTokenStore`/DB — слой связки с общим стором #49; логика проверки тестируется на моках без живого портала (как OAuth/invitation-ядро ранее). Cookie минта — `HttpOnly`+`Secure`+`SameSite=None`+**`Partitioned`** (CHIPS — primary-выбор для cross-site iframe, т.к. браузеры блокируют непартиционированные third-party cookies; fallback при несовместимости — токен через `postMessage`, не URL) — финализируется на привязке #49. |
| **OAuth-lifecycle hardening (миграция 0004) — тумбстоун + UPDATE-only + keep-alive-порог** | Устойчивость установки/удаления портала (improvement-plan §2). Два НЕЗАВИСИМЫХ гарда против воскрешения удалённого портала: (1) тумбстоун `portal_tombstone(member_id, deleted_ts=unix-сек)` — `save(tokens,{eventTs})` не пишет, если зафиксирован uninstall не старше события (B24 не гарантирует порядок online-событий → out-of-order install после uninstall); (2) `updateOnRefresh` — UPDATE-only (`RETURNING`): исчезла строка под uninstall → 0 строк → `false`, `accessToken` не отдаёт токен «мёртвого» портала. `deletePortal` (ONAPPUNINSTALL) — В ТРАНЗАКЦИИ: тумбстоун ПЕРЕД каскадным удалением данных портала в порядке зависимостей (FK на `portal(id)` — без `on delete cascade`; каскад только `response`→answer/product/insight, поэтому вручную). Keep-alive (`listNearExpiry`): порталы в полосе `[now-180д, now-177д)` по `updated_at` — refresh_token живёт ~180д, простаивающий портал иначе теряет токен; нижняя граница отсекает уже мёртвые гранты, кап батча 50. Отход от плана §2.2: единый `save(tokens, opts)` с opt-in-гардом вместо `saveOnInstall`/`updateOnRefresh`-разделения — обратно-совместимо (не трогает существующие вызовы). Известное ограничение (single-write, #4/§2.5): check-then-act в `save` и одиночный write в `updateOnRefresh` без межинстансного лока — при scale-out закрыть advisory-lock'ом. Keep-alive (§2.4) РЕАЛИЗОВАН (`server/plugins/keepalive.ts` + ядро `runKeepAlive`/`keepAliveIntervalMs`): каденция `TOKEN_KEEPALIVE_HOURS` (дефолт 24ч, клэмп [1ч,168ч] — верхний против overflow `setInterval`, где >2³¹ мс схлопывается в 1мс), гейт на `NUXT_B24_CLIENT_ID/SECRET`, немедленный первый прогон (переживает редеплои чаще каденции), `unref`. Лог-детали — в НЕ-`msg` полях (зарезервированный `msg` перетёр бы `fields.msg`). Отход от «Итога §2»: отдельный `keepalive.ts`, не единый `lifecycle.ts` (tombstone-sweep пока не написан). Обработка uninstall (§2.1) СДЕЛАНА: ветка `ONAPPUNINSTALL` в install-хендлере (`/api/b24/install` — Bitrix шлёт lifecycle-события на тот же handler URL, install-путь неизменён) → `decideUninstall` (constant-time сверка `application_token` — секрет app↔портал; берётся из ЗАШИФРОВАННОГО blob токенов `oauthTokensSchema`, поэтому колонка `portal.application_token` из 0004 ИЗБЫТОЧНА) → `deletePortal` только при `data.CLEAN=1` (respect выбора пользователя «очистить данные»; CLEAN=0 → сохраняем под переустановку). `parseBracketForm` (`bracket-form.ts`) разбирает форму B24 `auth[member_id]` в объект с гардом prototype-pollution. Формат события сверен с офиц. докой (`onAppUninstall`). Fail-closed: forged/unknown/повреждённый blob/нет стора → не удаляем (throw в удалении → error-лог + 200, B24 не ретраит). `CLEAN=0` (сохранить данные) НЕ пишет тумбстоун — устаревшие OAuth-токены остаются до переустановки (keep-alive затем логирует их фейлы ~несколько дней, пока не выпадут из near-expiry-окна) — осознанно, ради переустановки. `ts` капнут (`MAX_TS`) — иначе огромный `ts` через `greatest()` заблокировал бы переустановку тумбстоуном. member_id-binding (§2.3) СДЕЛАН: `verifyInstallMember` (`bitrix24/verify-install.ts`) при install рефрешит присланный refresh_token и сверяет AUTHORITATIVE `member_id` из ответа OAuth-сервера с заявленным (наш fetch-клиент `Bitrix24OAuth` отдаёт member_id из refresh напрямую, в отличие от SDK донора, где нужен сырой POST). Классификация по машинному `OAuthError.status`: **ТОЛЬКО 400 `invalid_grant`/401 или mismatch → 403** (подделка); **429 rate-limit/5xx/сеть/таймаут/пустой member_id → 503** (транзиент, fail-closed, не ложно-отвергаем; underlying-статус подмешан в `reason` для прод-логов). Рефреш РОТИРУЕТ токен → install сохраняет ВОЗВРАЩЁННЫЙ грант (чистая `applyVerifiedTokens`): свежие access/refresh, пересчёт `expiresIn`, сброс stale абсолютного `expires`; **`domain`/`memberId` — AUTHORITATIVE из гранта**, `clientEndpoint` ДЕРИВИТСЯ из `domain` (`https://<domain>/rest/`, клиент-присланный host НЕ используем → SSRF), `application_token` — сохранён из install-auth (рефреш его не возвращает). Присланный refresh мёртв; `verifiedAuth` идёт и в `save`, и в `registerIntegrations`. Синхронный верификационный рефреш обёрнут `AbortSignal.timeout` (10с, install-путь — не за деплой-таймаутом). Идемпотентность двойной доставки (page+event) — чистая `decideInstallDoubleDispatch(reason, portalExists)`: `refresh_rejected_*` + существующий портал → FINISH_HTML (гонка), иначе видимая ошибка (не маскируем мисконфиг); остаточный 1-бит install-оракул принят осознанно (нужен non-enumerable member_id, записи нет). **Хардеринг follow-up сделан**: `clientEndpoint` ВСЕГДА деривится из `domain` (`https://<domain>/rest/`; присланный/грант-endpoint как host не пробрасываются); `domain` валидируется `isAllowedPortalDomain` ДО REST на домен портала (400 иначе) → REST-host полностью покрыт; починена дыра регекса `isAllowedPortalDomain` (открытая «двойная TLD»-ветка пропускала `foo.bitrix24.evil.com` → SSRF, затрагивала и handshake #47) — TLD ужат до одного лейбла + явный `com.br`; `allowB24Install` (20/60с) ДО рефреша гасит амплификатор `oauth.bitrix.info`. Осталось по Фазе A: **живой smoke** (install page+event → токен captured → uninstall CLEAN=1/0); follow-up: `UNIQUE(domain)` (полное domain-poisoning), явный whitelist облачных TLD Bitrix (остаточный `bitrix24.<gTLD>`), общий стор лимитов + X-Forwarded-For (#4). |
| **Тёмная тема — `@nuxtjs/color-mode` (классовый `.dark`), по системе (#34)** | b24ui dark — классовый (`.dark` на `<html>`), не media. `@nuxtjs/color-mode` (preference `system`, `classSuffix: ''`) ставит этот класс по `prefers-color-scheme` — авто-тема для юзеров без тоггла, и детерминированный драйвер для гейта (`colorScheme`-проекты). Эмпирически: без `@nuxtjs/color-mode` класс `.dark` не проставлялся (тема не флипалась) — модуль регистрируем ДО b24ui. Свои подписи — `text-gray-500 dark:text-gray-400` (b24ui-компоненты адаптируются токенами сами). Гейт держит обе темы (`*-dark.png`). Ручной тоггл (#45) — нативный `B24ColorModeButton` (b24ui, глобально в `app.vue`): компонент построен поверх `@nuxtjs/color-mode` (тот же storageKey — конфликта нет), клик флипает preference light↔dark, иконка солнце/луна — чистый CSS по классу `.dark`. Сознательно 2-state (`B24ColorModeButton`), без явного «система»: во встроенном фрейме Bitrix24 достаточно explicit light/dark (вернуть «следовать системе» можно очисткой storage; `B24ColorModeSelect` с 3 состояниями — не нужен). aria-label кнопки — EN (локаль b24ui не настроена на ru; i18n — отдельно). Детерминизм гейта сохранён: на свежем контексте preference=system → тема по colorScheme-проекту; клик-путь проверяет отдельный тоггл-тест (без скриншота). При будущем CSP (#6) inline-скрипт color-mode (анти-FOUC) потребует nonce (`colorMode.script.nonce`) либо `'unsafe-inline'`. |
| **Persist прогресса — `localStorage["survey:${surveyKey}"]`, resume важнее deep-link (#34)** | Снимок (`SurveyFill.snapshot()`) кладётся per-опрос (ключ tenant-scoped, не `survey_progress` из прототипа — не конфликтует при нескольких опросах). Restore только на клиенте (`onMounted`, localStorage нет на SSR → визуальный гейт на fresh-контексте видит интро). Недоверенный снимок валидируется ядром (safeParse + сверка `surveyKey`/`versionNo`). При наличии снимка resume имеет приоритет над `?q` (не теряем прогресс). Снимок чистится на успешном submit; при ошибке submit намеренно сохраняется (повтор без потери ответов). Композабл-склейку тестируем Playwright'ом (e2e), не `@nuxt/test-utils` — вся логика снимка/restore/goTo уже под vitest-тестами `SurveyFill` (цена/польза). |
| **Vue-реактивность над `SurveyFill` — `shallowRef` + bump-тик (#34)** | `SurveyFill` (ядро) хранит состояние в обычном поле и меняет его по ссылке — Vue это не отслеживает. Композабл `useSurvey` держит `shallowRef(fill)` + счётчик `tick`; `view` (computed) зависит от `tick`, каждая мутация сопровождается `bump()`. Почему так, а не реактивный объект внутри ядра: ядро остаётся framework-agnostic (без зависимости от Vue `reactive`), реактивность — ответственность Vue-слоя. Цена: дисциплина `bump()` после каждой мутации (зафиксировано в JSDoc композабла как предупреждение). Загрузка версии — на странице через `useAsyncData` (SSR-payload + рефетч при смене `:key`), не внутри композабла. |
| **Nuxt-приложение в `app/`, ядро в `src/` (единый пакет, не монорепо)** | Каркас контура A живёт в `app/` (Nuxt `srcDir`), ядро остаётся в `src/` и Nuxt'ом не сканируется; доступ к ядру — алиас `~core`. Единый `package.json`/lockfile (не монорепо) — проще CI/деплой, ядро и UI версионируются вместе. Ядровой `pnpm check` независим от Nuxt (root tsconfig типизирует только `src/test/scripts`); Nuxt-сборка/типы — отдельно. Это сохраняет «framework-agnostic ядро» как принцип: UI зависит от ядра, не наоборот. |
| **Визуальный гейт #13 — fixture-first, Stop-хук, пока не в CI** | Гейт (Playwright скриншот-регрессия) подняли ДО экранов на фикстуре-заглушке: машинерия доказана end-to-end, реальные экраны сразу под гейтом. Энфорсмент — Stop-хук (`.claude/hooks/visual-gate.sh`) в предустановленной среде (chromium в `/opt/pw-browsers`), а не блокирующий CI: разница шрифтов/версий браузера между средами даёт ложные провалы, пока нет закреплённого контейнера. Узкий триггер (чистое ядро не трогает) + мягкая деградация без браузера. Детерминизм: reduced-motion, system-шрифты, фикс-вьюпорты-брейкпоинты. См. `docs/visual-gate.md`. |
| **Brain-first** | Продаём не анкеты, а решения: AI-слой превращает ответы (вкл. фритекст) в число (CSAT/NPS) + причину + действие. Опрос — самый дешёвый датчик, защищённый ров — AI-анализ. См. `brief.md` §13. |
| **`invitationPolicy` — version-frozen (#21)** | Политика приглашений вшита в схему/версию, а не в рантайм-конфиг: ответ должен интерпретироваться по той версии, на которой собран. Иммутабельность версии распространяется и на политику. |

| **Локальное приложение (1 портал = 1 инстанс)** | Решение №7: своя БД на сервере клиента, без кросс-портальной мультитенантности — проще изоляция и приватность; tenant-ключ `portalId`. |
| **Durable-идемпотентность ответа — по invitation_token, не по id записи** | `id` ответа сервер генерит на каждый submit (ретрай дал бы новый) — негодный ключ дедупа. Якорь — `invitation_token`: частичный UNIQUE `(portal_id, invitation_token) where invitation_token is not null` (миграция 0003).
