# Индекс решений

> Короткое «почему так» по ключевым архитектурным решениям. Продуктовые решения
> (№1–8) — в [`brief.md`](./brief.md) §12; здесь — технические решения ядра.
> Принцип: каждое нетривиальное решение фиксируется (конвенция `CLAUDE.md`).

| Решение | Почему так |
|---|---|
| **Тёмная тема — `@nuxtjs/color-mode` (классовый `.dark`), по системе (#34)** | b24ui dark — классовый (`.dark` на `<html>`), не media. `@nuxtjs/color-mode` (preference `system`, `classSuffix: ''`) флипает класс по `prefers-color-scheme` — авто-тема для юзеров без тоггла, и детерминированный драйвер для гейта (`colorScheme`-проекты). b24ui ждёт именно `@nuxtjs/color-mode` (`hasNuxtModule`), без него dark не активируется. Свои подписи — `text-gray-500 dark:text-gray-400` (b24ui-компоненты адаптируются токенами сами). Гейт держит обе темы (`*-dark.png`). |
| **Persist прогресса — `localStorage["survey:${surveyKey}"]`, resume важнее deep-link (#34)** | Снимок (`SurveyFill.snapshot()`) кладётся per-опрос (ключ tenant-scoped, не `survey_progress` из прототипа — не конфликтует при нескольких опросах). Restore только на клиенте (`onMounted`, localStorage нет на SSR → визуальный гейт на fresh-контексте видит интро). Недоверенный снимок валидируется ядром (safeParse + сверка `surveyKey`/`versionNo`). При наличии снимка resume имеет приоритет над `?q` (не теряем прогресс). Снимок чистится на успешном submit; при ошибке submit намеренно сохраняется (повтор без потери ответов). Композабл-склейку тестируем Playwright'ом (e2e), не `@nuxt/test-utils` — вся логика снимка/restore/goTo уже под vitest-тестами `SurveyFill` (цена/польза). |
| **Vue-реактивность над `SurveyFill` — `shallowRef` + bump-тик (#34)** | `SurveyFill` (ядро) хранит состояние в обычном поле и меняет его по ссылке — Vue это не отслеживает. Композабл `useSurvey` держит `shallowRef(fill)` + счётчик `tick`; `view` (computed) зависит от `tick`, каждая мутация сопровождается `bump()`. Почему так, а не реактивный объект внутри ядра: ядро остаётся framework-agnostic (без зависимости от Vue `reactive`), реактивность — ответственность Vue-слоя. Цена: дисциплина `bump()` после каждой мутации (зафиксировано в JSDoc композабла как предупреждение). Загрузка версии — на странице через `useAsyncData` (SSR-payload + рефетч при смене `:key`), не внутри композабла. |
| **Nuxt-приложение в `app/`, ядро в `src/` (единый пакет, не монорепо)** | Каркас контура A живёт в `app/` (Nuxt `srcDir`), ядро остаётся в `src/` и Nuxt'ом не сканируется; доступ к ядру — алиас `~core`. Единый `package.json`/lockfile (не монорепо) — проще CI/деплой, ядро и UI версионируются вместе. Ядровой `pnpm check` независим от Nuxt (root tsconfig типизирует только `src/test/scripts`); Nuxt-сборка/типы — отдельно. Это сохраняет «framework-agnostic ядро» как принцип: UI зависит от ядра, не наоборот. |
| **Визуальный гейт #13 — fixture-first, Stop-хук, пока не в CI** | Гейт (Playwright скриншот-регрессия) подняли ДО экранов на фикстуре-заглушке: машинерия доказана end-to-end, реальные экраны сразу под гейтом. Энфорсмент — Stop-хук (`.claude/hooks/visual-gate.sh`) в предустановленной среде (chromium в `/opt/pw-browsers`), а не блокирующий CI: разница шрифтов/версий браузера между средами даёт ложные провалы, пока нет закреплённого контейнера. Узкий триггер (чистое ядро не трогает) + мягкая деградация без браузера. Детерминизм: reduced-motion, system-шрифты, фикс-вьюпорты-брейкпоинты. См. `docs/visual-gate.md`. |
| **Brain-first** | Продаём не анкеты, а решения: AI-слой превращает ответы (вкл. фритекст) в число (CSAT/NPS) + причину + действие. Опрос — самый дешёвый датчик, защищённый ров — AI-анализ. См. `brief.md` §13. |
| **`invitationPolicy` — version-frozen (#21)** | Политика приглашений вшита в схему/версию, а не в рантайм-конфиг: ответ должен интерпретироваться по той версии, на которой собран. Иммутабельность версии распространяется и на политику. |
| **Презентация (`intro`/`thanks`/`blocks`) — version-frozen (#25)** | Контент экранов Интро/Спасибо и имена блоков вшиты в `SurveyDraft`/`CompiledVersion` и едут в `compiled_schema` — единый источник правды для Vue-слоя (а не второй источник/хардкод), заморожен со снимком анкеты. `blocks` — упорядоченный массив имён (минимально), `question.block` остаётся именем; `SurveyFill` не трогает (он про прохождение). |
| **AES-256-GCM с `kid` в blob** | Шифрование `portal.tokens` с идентификатором ключа (`kid`) внутри blob → форвард-совместимость ротации ключа без миграции данных. `loadTokenKey` — startup-guard (падает на пустом/плейсхолдерном ключе). См. `bitrix24/crypto.ts`. |
| **Keyset-пагинация (не OFFSET)** | Курсор по стабильному ключу даёт стабильную выборку при вставках и постоянную стоимость страницы (OFFSET деградирует на хвосте). Helpers — `store/cursor.ts`. |
| **Подавление малых N в слое чтения** | Анонимность гарантируется там, где данные отдаются (PgStore/read-API), а не в «сырых» агрегатах — нельзя обойти, читая агрегат напрямую. |
| **Схема — единый источник истины (zod)** | Типы и runtime-валидация выводятся из одних zod-схем → TS-тип и проверка не расходятся. `domain/schema.ts`. |
| **Драйвер-агностичный PgStore** | `Queryable` ≈ `pg.Pool`/pglite: те же `.sql`-миграции применяют и прод-Postgres, и pglite-тесты (единый источник схемы, паритет с in-memory). |
| **Logger — интерфейсо-зависим, не вендоро** | Ядро знает интерфейс `Logger`, прод подменяет адаптером (Pino/Sentry) инъекцией — zero-dep ядро, без вендор-лока. См. `docs/observability.md`. |
| **Локальное приложение (1 портал = 1 инстанс)** | Решение №7: своя БД на сервере клиента, без кросс-портальной мультитенантности — проще изоляция и приватность; tenant-ключ `portalId`. |
| **Durable-идемпотентность ответа — по invitation_token, не по id записи** | `id` ответа сервер генерит на каждый submit (ретрай дал бы новый) — негодный ключ дедупа. Якорь — `invitation_token`: частичный UNIQUE `(portal_id, invitation_token) where invitation_token is not null` (миграция 0003). Повтор приглашения на любом инстансе → `ON CONFLICT DO NOTHING` (durable single-use поверх in-memory `MemoryInvitationStore`, закрывает мульти-инстанс-щель #4). Ответы по публичной ссылке без приглашения дедупу не подлежат (токена нет). Токен хранится в колонке открытым — приемлемо, т.к. он одноразовый и после расхода мёртв; тех-долг — миграция 0004 переключит ключ на FK `response.invitation_id`, когда invitation-flow получит общий стор (#4), и токен из `response` уйдёт. `invitationToken` редактируется логгером (правило подстроки `token`) и должен strip'аться из любой публичной проекции `ResponseRecord` (требование #31). |
| **Публичная проекция версии — в HTTP-хендлере, не в сторе/схеме** | `GET /api/survey/:key/current` отдаёт версию БЕЗ `invitationPolicy` (триггер-стадии/канал — внутренняя CRM-конфигурация). Отсечение в `toPublicVersion` (api-слой), а не в `IStore`/схеме: стор и аналитика политику видят, наружу не утекает только публичный read-контур. Тип `Omit<…,'invitationPolicy'>` гейтит будущие чувствительные поля компилятором. |
