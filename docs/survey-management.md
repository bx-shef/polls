# Управление опросами — как это устроено и что делать

Практический документ: где живут опросы, как добавить вопрос, как почистить данные,
куда ещё можно встроить виджеты, и честная рефлексия о сложности аналитики.

## Как устроены опросы (модель)

Опрос = **черновик** (`SurveyDraft`) → `compile()` → **иммутабельная версия** (`store.publish(draft, N)`).
- У каждого вопроса/опции **стабильный ключ** (`question_key`/`option_key`) — якорь сопоставимости
  аналитики между версиями (изменил формулировку — история не ломается).
- Опубликованную версию **нельзя переписать** (только выпустить новую: v2, v3…).
- Презентация (интро/«спасибо»/блоки) и `invitationPolicy` (триггер-стадии) **вшиты в версию**
  (version-frozen) — снимок на момент публикации.

### Привязка опроса к сущности и направлению

`invitationPolicy` декларирует **датчик запуска** опроса:
- `entityType` — тип сущности Bitrix24: `deal` (сделка, дефолт), `lead`, `spa` (смарт-процесс),
  `contact`, `company`, `task` (задача, модуль задач — **вне CRM-воронок**; статусы задачи кладутся
  в `triggerStages`). Старые опросы без поля трактуются как «по сделке».
- `spaEntityTypeId` — числовой id смарт-процесса (`crm.item`), когда `entityType === 'spa'`.
  **Обязателен** при `entityType=spa` (черновик без него не пройдёт `compile()` — `superRefine`);
  для прочих типов поле **запрещено** (схема отвергает — жёсткий инвариант, не рекомендация).
- `triggerStages` — стадии/статусы этой сущности (портал-специфичны), переход в которые запускает
  опрос. Это и есть привязка к **направлению**: стадии живут в конкретной воронке = направление.

На каждое направление и тип сущности можно завести **1 или несколько** опросов: `surveysTriggeredBy(stageId)`
возвращает массив ключей опросов, чья текущая версия триггерится этой стадией (GIN-индекс по
денормализованным `trigger_stages`, #22). Маппинг сущность→контекст и плейсменты в карточках
лида/смарт-процесса/контакта — фаза «мульти-сущность» (см. «Дальше»); задача — ручной запуск реализован (`TASK_VIEW_SIDEBAR`, `task.ts`), автотриггер по статусу — нет.

Источник истины типов/валидации — `src/domain/schema.ts` (zod). Обезличенный шаблон валидного
черновика — `docs/reference/survey-schema.template.json`.

## Где сейчас задаются опросы

✅ **Конструктор опросов есть** (`app/pages/admin/surveys/`): экран-список с фильтром по сущности +
конструктор (правка заголовка/стадий-триггеров; добавить/удалить/переставить вопросы и опции;
тип/метрика/обязательность/баллы → публикация новой иммутабельной версии). Drag-and-drop порядком
мыши — фаза полировки (пока кнопки ↑/↓).
Опрос — это `SurveyDraft` (демо — `src/demo/seed.ts`, `draftV1/draftV2`), публикуется через `store.publish`.

**Модель без таблицы черновиков:** в сервисе НЕТ отдельного draft-хранилища. Черновик живёт в
редакторе (клиенте); «сохранить» = опубликовать НОВУЮ иммутабельную версию. `GET /api/admin/surveys/:key`
восстанавливает черновик из текущей версии через обратную проекцию `versionToDraft` (без versionNo/
compiledAt, сохраняя `invitationPolicy`), а `POST …/publish` создаёт версию `currentVersionNo+1`.
Незавершённые правки при закрытии вкладки теряются — приемлемо для текущей фазы (draft-таблица — если появятся
соавторы/preview).

Способы завести/обновить опрос сегодня:
1. **Через Admin API** (под `requirePortalSession`): `GET /api/admin/surveys/:key` → редактор →
   `POST /api/admin/surveys/:key/publish` (тело — `SurveyDraft`, номер версии назначает сервер).
   Это рабочая точка для будущего визуального конструктора.
2. **Через сид** (демо/single-tenant): отредактировать `draftV*` в `src/demo/seed.ts` → пересборка/деплой.
3. **Через стор** (программно): `await store.publish(draft, versionNo)` — на любом `IStore`
   (Memory/Pg). Это же делает `seedDemoIfEmpty` при первом старте на пустой БД.

## Точка входа админ-UI (список опросов)

Список опросов портала — `GET /api/admin/surveys` (auth: `requirePortalSession`, fail-closed):
возвращает `{ ok, surveys: SurveySummary[] }` — по каждому опросу ключ, заголовок, язык, номер
текущей версии и привязку-датчик (`entityType`/`spaEntityTypeId`/`triggerStages`). Это основа
экрана-списка с фильтром по сущности/направлению (макет на основе шаблонов печатных форм Bitrix24).
Tenant-scoped внутри PgStore по `portalId`; мульти-тенант + rate-limit — #49.
Редактирование опроса — `GET /api/admin/surveys/:key` (текущая версия как черновик через
`versionToDraft`) + `POST /api/admin/surveys/:key/publish` (публикация новой версии). Сам визуальный
полноценный конструктор (список + add/remove/reorder вопросов и опций) уже реализован (`app/pages/admin/surveys/`); drag-and-drop порядком мыши — «Дальше».

## Как добавить новый вопрос

1. В черновике (`SurveyDraft.blocks[].questions[]`) добавить вопрос с **новым стабильным
   `question_key`** (латиница/цифры/`_`), типом (`single`/`multi`/`text`/шкала) и метрикой
   (`nps`/`csat`/`ces`/`none`).
2. Выпустить **новую версию**: `store.publish(updatedDraft, prevVersion + 1)`.
3. Старые ответы остаются на своих версиях; аналитика сопоставляет ряды по `question_key`
   (новый вопрос появляется с версии публикации — тренд версионно-безопасен).

Метрика вопроса (`nps`/`csat`/…) определяет, в какой расчёт он попадёт — дашборд берёт вопросы
**по метрике из текущей версии** (не по хардкод-ключам).

## Как удалить данные за прошлый период

Данные ответов — таблица `response` (PostgreSQL), tenant-scoped по `portal_id`. Удаление за период:

```sql
-- ВНИМАНИЕ: необратимо. Только нужный портал и период. Сделать бэкап.
delete from response
 where portal_id = (select id from portal where member_id = '<member_id портала>')
   and submitted_at < '2026-01-01T00:00:00Z';
```
- `response_product` (товары ответа) удалится каскадом (FK).
- Версии опроса (`survey_version`) не трогаем — они нужны для рендера/сопоставимости.
- Полная очистка демо-данных портала: `delete from response where portal_id = …` (все ответы).

Сейчас это **ручная операция** (психология «ядро + деплой»); кнопка «очистить период» в дашборде/
админке — отдельная задача (в issue). На уровне приложения данные не теряются сами (durable, volume `db-data`).

## Куда ещё можно встроить виджеты (точки интеграции)

Реализовано (регистрируется при установке, см. `bitrix24-integration.md`): робот `bizproc.robot.add`,
плейсменты `CRM_DEAL_DETAIL_ACTIVITY` (карточка сделки), `CRM_ANALYTICS_MENU` (меню аналитики)
и `TASK_VIEW_SIDEBAR` (карточка задачи — ручной запуск, `task-widget.vue`).

**Маршрутизация по сущности** (env, временный слой): какой опрос запускает виджет из карточки,
задаётся переменными `SURVEY_KEY_DEAL`, `SURVEY_KEY_TASK` и т.д. (полный список — `.env.example`),
по умолчанию — `csat_postdeal`. Постоянный UI-маппинг в конструкторе — отдельный issue (`docs/issues.md`).

Дополнительные точки (по докам Bitrix24 `placement.bind`) — на будущее, под охват/удобство:
- **`CRM_DEAL_DETAIL_TAB`** — полноценная вкладка в карточке сделки (больше места, чем виджет активности):
  показать историю опросов/результат прямо в сделке.
- **`CRM_LEAD_DETAIL_ACTIVITY` / `CRM_CONTACT_*` / `CRM_COMPANY_*`** — тот же виджет запуска опроса
  на лидах/контактах/компаниях (не только сделки).
- **`LEFT_MENU`** — приложение пунктом в левом меню портала (общий вход в «Опросы»).
- **`crm.activity.configurable.add` / таймлайн** — запись результата опроса в ленту сделки (направление
  «результат → CRM», #18) + `crm.automation.trigger.add` для продвижения сделки по итогу.
- **`imbot` (чат-бот)** — доставка ссылки на опрос сообщением в Битрикс24 (канал доставки).

Эти точки НЕ требуют переделки ядра — добавляются как новые `placement.bind`/обработчики поверх
готового `client.ts` (b24jssdk). Заводятся в issue по мере приоритета.

## Рефлексия: не слишком ли сложная аналитика?

Честный вопрос «может, не стоит так заморачиваться — никто не оценит». Разбор:

**Что точно нужно (клиент это видит и за этим пришёл):**
- **NPS / CSAT** по опросу — основная цифра.
- **Срезы по ответственному и услуге/направлению** — «кто/что тянет вниз». Это и есть управленческая
  ценность ради которой ставят сервис.

**Что «дорого выглядит», но дёшево и оправдано (уже построено, не выкидываем):**
- **Подавление малых выборок (k-анонимность)** — не «академизм», а защита от конфуза: без него дашборд
  покажет «NPS по сотруднику Иванову (n=1)» = по сути оценку конкретного человека одним клиентом.
  Один такой случай у заказчика — репутационный удар. Стоит копейки, предотвращает дорогое.
- **Версионно-безопасный тренд** — без него изменение анкеты ломает график и подрывает доверие к цифрам.

**Где НЕ стоит усложнять (вывод):**
- **UI должен показывать то, на что реагируют:** NPS/CSAT + топ-срезы (ответственный/услуга) на первом
  экране. Глубокие срезы (клиент, направление, помесячный тренд, фильтр по версии) — **прогрессивное
  раскрытие** (вкладки/«показать детали»), а не всё сразу. Ядро уже всё считает — вопрос подачи, не расчёта.
- **Не добавлять новые метрики «потому что можем».** Текущего набора (NPS/CSAT/CES/распределения)
  хватает; следующий шаг ценности — не ещё метрика, а **связка с Bitrix** (датчик→опрос→KPI в Ленте).

Итог: расчётное ядро оставляем как есть (гарантии приватности/корректности дёшевы и уже под тестами),
а инвестируем в **подачу** (что показать на первом экране) и **связку с порталом**, а не в новые формулы.
