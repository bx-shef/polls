# Правила работы с репозиторием

## Язык

Общаемся на русском. Документацию пишем на русском.

## Ветки и пуши

- **В `main` напрямую не пушим никогда.**
- Вся работа ведётся в отдельной ветке, изменения вносятся через Pull Request.
- Так как репозиторий приватный, PR описываем текстом прямо в теле PR — что сделано и зачем.

## Слияние

- **Мержит всегда владелец репозитория сам.** Автомерж и принудительный мерж не используем.
- Перед мержем PR проверяется: нет ли конфликтов, корректны ли изменения, не затронуто ли лишнее.

## Pull Request

- Описание PR пишем на русском: что изменено, почему, на что влияет.
- В один PR — одна логическая задача. Не мешаем несвязанные правки.
- PR не мержится пока не получен явный сигнал от владельца.

## База знаний и оповещения

База знаний (`docs/`).
- После каждого смерженного PR синхронно обновляем `docs/` —
  документация не должна отставать от реального состояния.

## Отчёты и навыки

- Промпты отчётов живут в `docs/reports/` (**канон**), а их зеркало — навыки в
  `.claude/skills/` (`/report-status`, `/report-digest`, `/report-questions`).
- При правке промпта в `docs/reports/` **синхронно** обновляй соответствующий
  `SKILL.md`, и наоборот. Идентичность тел проверяет `scripts/check-skills.sh`
  (linux) / `scripts/check-skills.ps1` (windows) и CI.

## GitHub API Rate Limits

Квоты считаются раздельно: **REST-core** (5000 **запросов**/час) и **GraphQL**
(5000 **очков**/час — points, не запросы). MCP-инструменты GitHub для записи/поиска/листинга
(`issue_write`, `search_issues`, `list_issues`, резолв `duplicate_of`) ходят через **GraphQL**;
один вложенный list/search стоит >1 очка, поэтому GraphQL выгорает первой, а REST-core остаётся
свободен.

- **Сначала смотри, какая квота кончилась**, не жди вслепую: `curl -s -H "Authorization: Bearer
  $GITHUB_TOKEN_INGEST" https://api.github.com/rate_limit` отдаёт `remaining`+`reset` (epoch) по
  каждому пулу — это **разные** лимиты: `core`, `graphql`, `search`. Поллер на этом эндпоинте
  бесплатен. Жди до нужного `reset` **+2–5 с** (запас на рассинхрон часов).
- **Читай прямым REST**, где можно: `GITHUB_TOKEN_INGEST` (read-only, не трогает GraphQL) — для
  диагностики/enumeration. Для перебора бери постранично `GET
  /repos/{o}/{r}/issues?state=all&per_page=100` (100 issue = 1 запрос) вместо одиночных `GET
  /issues/{n}` (100 запросов); single-GET — только когда нужен конкретный номер.
- **Кэшируй повторные чтения** условными запросами: `If-None-Match: <etag>` → `304` квоту не
  тратит. Только REST; в GraphQL ETag нет.
- **Записи (close/comment/labels/duplicate) — только через MCP** (GraphQL): **батчи** их в один
  заход и **не молоти** `list/search` в цикле — они тоже жрут GraphQL-очки и роняют последующую
  запись.
- **Помни про secondary limits** (отдельный механизм поверх primary, отдают `403`): даже при
  свободной primary-квоте контент-операции режутся на ~**80/мин и 500/час**, плюс ≤100
  одновременных запросов (общий пул REST+GraphQL) и ≤2000 очков/мин на GraphQL. Поэтому записи
  разноси во времени + exponential backoff с jitter, а не лей пачкой.
- `GITHUB_TOKEN_INGEST` — **read-only**: writes через него дают `Resource not accessible`.

## Bitrix24 — настройка и проверка через хук

При работе с Bitrix24 и его настройкой заведи переменную `B24_HOOK___SUFFIX` —
входящий вебхук портала с **максимальными правами** (`___SUFFIX` — твой суффикс под
портал/назначение, подставляешь сам: `B24_HOOK_MAIN`, `B24_HOOK_DEV`, …). С таким хуком
агент не только читает, но и **настраивает и проверяет** Bitrix24: создаёт и правит
сущности, меняет настройки портала и тут же верифицирует результат через REST.

> [!CAUTION]
> Хук с максимальными правами — это «ключ от всего портала», секрет. Держи его в
> окружении/секретах (не в git, не в логах), используй для setup- и админ-задач. Для
> постоянного рантайма заводи **отдельный** вебхук с минимально необходимыми правами.


# Типовой сценарий для review

```
Если ты еще не открыл PR на свою работу - открой.

Забери изменения из ветки main.

Поясни простым языком о чем PR.

Потом /review проверь (с разных сторон).

Так же параллельно привлеки
(скажи агентам что проект большой и проверять нужно так что бы по таймауту не упасть):

1) LLM (модель Sonnet) специалиста на оценку документации и файлов Skill.

2) LLM (модель Sonnet) программиста - оценить адекватность принятых решений, покрытие комментариями jsDocs , типизации (TS) и другие аспекты на его усмотрение

3) LLM (модель Sonnet) тестировщика - оценить покрытие тестами и сами тесты
.
4) LLM (модель Sonnet) отдел безопасности.

5) LLM (модель Sonnet) тех-директора.

Отчет о проверке покажи (на русском): краткий - кто, что, почему, как исправить.

И потом все начинай править - все правим в этом PR - если точно уверен что нужно вынести правки в ISSUE или PR, то давай обсуждать это.
```

# Типовой сценарий для merge

```
Давай готовится к закрытию PR.

Перед мержем:
— проверь обновления из ветки main;
— проверь CI;
— проверь что закрыты все вопросы review;
— follow-up issues - на английском языке заводи как issue или расширяй текущие (контекст хороший делай);
— напоминание про squash-сообщение - если были BREAKING CHANGE - подумай как правильно это обыграть при закрытии что бы потом мы могли написать что менялось в Changelog;
— освежить Last reviewed штампы (доки, skill) на дату мержа.

Если все хорошо — подготовь PR к мержу и доложи. Мерж выполняет владелец сам
(см. раздел «Слияние»); агент не мержит и не удаляет ветки без явной команды.

После мержа владельцем (по его команде):
— ветки удаляй

Если PR закрыл вопрос issue нужно написать сообщение:
— на английском, бодрое, в меру шутливое;
— пару примеров или ссылок на документацию;
— если приемлемо, то добавить стоит пример промпта для использования;
— спасибо от меня.

И если есть  возможность - закрываем issue.

Потом оцени проект:
— кратко что сделано, какой следующий шаг и что потом;
— отдельно что мешает;
— пиши простыми и понятными словами.
```
