## 1. Что это и зачем

**Продукт.** Анонимный одностраничный опрос-исследование. Тема прототипа —
«Как живут российские вайбкодеры» (AI-инструменты, проекты, факапы). Сам движок
**не привязан к теме**: весь контент задаётся декларативно в JSON-схеме, поэтому
платформа переиспользуема под любой опрос.

**Ключевые свойства:**

| Свойство | Значение |
|---|---|
| Анонимность | Без персональных данных, без авторизации |
| Длительность | ~5–7 минут |
| Объём | 25 вопросов, 8 тематических блоков |
| Типы вопросов | Один вариант (`single`) и несколько (`multi`) |
| Спец-механики | «Другое + свой текст», взаимоисключающие варианты, необязательные вопросы |
| Сохранение прогресса | `localStorage`, можно вернуться и дозаполнить |
| Защита от повтора | После отправки опрос повторно недоступен |
| Результат | Открытое исследование (агрегированные данные) |

**Цель брифа** — точно воспроизвести этот UX и контент на стеке
`templates-dashboard` (Nuxt 4 + Bitrix24 UI) и развернуть его по
производственному паттерну из `templates-mcp` (Docker + reverse-proxy + TLS).

## 2. Архитектура референс-прототипа

Прототип — **статический SPA без сборки**, отдаётся nginx. Четыре артефакта:

```
index.html            // каркас 3 экранов: #intro / #survey / #thanks
survey.css            // дизайн-токены и компоненты (~665 строк)
survey-app.js         // вся логика, vanilla JS, IIFE (~432 строки)
survey-schema.json    // контент: интро, блоки, 25 вопросов, тексты финала
```

Плюс два серверных эндпоинта (см. §8): `GET /api/session`, `POST /api/submit`.

**Принцип:** схема грузится через `fetch('./survey-schema.json')`, затем
`boot()` строит состояние и рендерит вопросы по одному. Контент полностью
отделён от логики — менять опрос можно без правки кода.

Верифицированный слепок схемы сохранён в
[`docs/reference/survey-schema.json`](./reference/survey-schema.json) — это
источник истины по всем формулировкам и вариантам ответов.

**Состояние (в памяти + `localStorage`):**

```js
answers = [ { picked: [/* индексы выбранных опций */], other: "" }, … ] // по одному на вопрос
current = 0                                                              // индекс текущего вопроса
```

- `localStorage["vibesurvey_progress"]` — снапшот `answers` (пишется на каждое
  изменение) → возобновление сессии.
- `localStorage["vibesurvey_done"]` + cookie `vibesurvey_done=1` (срок 1 год) →
  блокировка повторного прохождения.

## 3. Пользовательский поток и протокол стадий

Три экрана-секции переключаются классом `.is-active`; на теле `#app` —
модификатор `.survey-on` (показывает верхнюю полосу прогресса).

```
ИНТРО ──[Начать опрос]──▶ ОПРОС (вопрос 1 … 25) ──[Отправить]──▶ СПАСИБО
                              ▲              │
                              └─ Назад ◀─────┘
```

### 3.1. Экран ИНТРО (`#intro`)

Что показано:
- Вордмарк **«Вайбкодинг.»** (точка — акцентным цветом) + метка **«Опрос '26»**.
- Кикер: «Первое большое исследование».
- Заголовок: «Как живут российские вайбкодеры?» (в 3 строки, `white-space: pre-line`).
- Лид-абзац с описанием.
- Три «чипа»: **Анонимно** · **~5–7 минут** · **Без персональных данных**.
- CTA-кнопка **«Начать опрос»** + подпись «25 вопросов · 8 блоков».
- Скрытый honeypot-input `#hp` (анти-бот, см. §8).

Переход по клику CTA → `startSurvey()`: с `#intro` снимается `is-active`,
`#survey` получает `is-active`, `#app` получает `survey-on` (появляется
топ-полоса), `current = 0`, рендер первого вопроса.

### 3.2. Экран ОПРОС (`#survey`)

**Постоянный «хром» (не зависит от вопроса):**

- **Топ-полоса прогресса** (`#topfill`) — ширина `= (current+1)/25 × 100%`.
- **Левый рейл (только десктоп, 38% ширины):** бренд, имя блока (`#railBlock`),
  крупный номер `#railNum` «01 / 25», прогресс-бар (`#railFill`), счётчик
  `#railCount` «Вопрос N из 25», подсказки клавиш внизу.
- **Мобильный хедер (`#mhead`, только мобайл, sticky):** имя блока (`#mBlock`) +
  номер `#mNum` «01 / 25».

**Контент вопроса (рендерится в `#stage` на каждый шаг):**

1. `q__meta` — «01 / 25 — ИМЯ БЛОКА» (на мобайле скрыт, дублируется в хедере).
2. `q__title` — текст вопроса.
3. `q__helper` — подсказка: **«Один вариант»** (single) / **«Можно выбрать
   несколько»** (multi) с иконкой; для необязательных — тег **«необязательно»**.
4. `opts` — кнопки-варианты (radio/checkbox), опционально в 2 колонки; у первых
   девяти — хинт клавиши `1`–`9`; у «Другое» — раскрывающееся текстовое поле.
5. `nav` — **Назад** (выключена на Q1) · хинт «Enter далее» · **Далее**
   (на последнем вопросе — **Отправить**).

**Что обновляется при каждом переходе (`updateChrome()` + `render()`):**

| Элемент | Обновление |
|---|---|
| `#topfill`, `#railFill` | ширина прогресса в % |
| `#railNum` | «NN / 25» |
| `#railBlock`, `#mBlock` | имя текущего блока |
| `#railCount` | «Вопрос N из 25» |
| `#mNum` | «NN / 25» |
| `#stage` | новый вопрос с анимацией `qin` (fade + сдвиг 6px, 0.2s) |
| прокрутка | сбрасывается в начало (`scrollTop`) |

**Границы блоков.** Имя блока (`railBlock`/`mBlock`/`q__meta`) меняется при
пересечении границ блоков — соответствие «вопрос → блок» см. §4.

### 3.3. Экран СПАСИБО (`#thanks`)

После успешной отправки: с `#survey` снимается `is-active`, с `#app` —
`survey-on` (топ-полоса скрывается), `#thanks` получает `is-active`. См. §6.

### 3.4. Навигация и клавиатура

- **Кнопки:** Назад / Далее(Отправить). На Q1 «Назад» выключена.
- **Клавиатура (десктоп):** `1`–`9` — выбрать вариант; `Enter` или `→` — далее;
  `←` — назад. Блокируется, пока фокус в текстовом поле «Другое».
- **Валидация при «Далее»:** если вопрос обязателен и ничего не выбрано —
  переход блокируется, `q__helper` краснеет (`is-error`), текст меняется на
  «Выберите вариант» (single) / «Выберите хотя бы один вариант» (multi).
  Необязательные (блок 8) пропускаются без выбора.
- **Deep-link для превью:** `?q=N` или `#qN` (N = 1…25) открывает опрос сразу
  на нужном вопросе — удобно для ревью макетов.

  Выбор заменяет предыдущий.
- **`multi` (7 вопросов):** несколько вариантов, индикатор-«чекбокс» (галочка).
  `q_users`, `q_replaced`, `q_breaks`, `q_hardest`, `q_devs_attitude`,
  `q_tools`, `q_models`.
- **`columns: 2` (5 вопросов):** двухколоночная сетка вариантов на десктопе
  (`q_replaced`, `q_breaks`, `q_hardest`, `q_tools`, `q_models`). На мобайле
  всегда 1 колонка.
- **`exclusive` (3 опции):** взаимоисключающий вариант. При его выборе снимаются
  все прочие; при выборе обычного — снимается эксклюзивный. Это «Пока нет
  работающего проекта» (`q_users`), «Ничего не заменил, только дополнил»
  (`q_replaced`), «Отношение не изменилось» (`q_devs_attitude`).
- **`other` (7 вопросов):** вариант «Другое» со своим текстовым полем — см. §5.
- **`required: false` (4 вопроса, блок 8):** демография; можно пропустить,
  «Далее» не блокируется, пустые ответы на сервер **не отправляются**.

> Полные формулировки всех вопросов и вариантов — в
> [`docs/reference/survey-schema.json`](./reference/survey-schema.json).
> При переносе тексты берём оттуда дословно.

## 5. «Другое — свой вариант» (отдельный разбор)

Опция-«Другое» помечена в схеме флагом `"other": true` (всегда `id: "other"`,
`label: "Другое"`). Встречается в 7 вопросах: `q_role`, `q_replaced`,
`q_reason`, `q_breaks`, `q_hardest`, `q_tools`, `q_models`.

**Разметка и поведение:**

1. Рендерится как обычная кнопка-вариант, но **сразу за ней** — скрытый
   раскрывающийся контейнер `.other-wrap[data-for=i]`.
2. При выборе «Другое» контейнеру добавляется `.is-open`; CSS анимирует
   `grid-template-rows: 0fr → 1fr` (0.25s), внутренник с `overflow: hidden` —
   поле плавно «выезжает».
3. Внутри `.other-field`: текстовый `input#otherinput`
   (`maxlength=80`, `placeholder="Свой вариант…"`) и счётчик символов
   `#othercount` формата **«X/80»**.
4. **Авто-фокус**: при выборе «Другое» поле получает фокус через 180 мс.
5. Значение хранится в `answers[current].other` и пишется в `localStorage` на
   каждое нажатие клавиши.

**Логика в зависимости от типа вопроса:**

- В `single`-вопросе выбор «Другое» (как любой вариант) снимает прочие — остаётся
  только «Другое» + текст.
- В `multi`-вопросе «Другое» сосуществует с другими (с учётом правил
  `exclusive`).
- При **снятии** выбора «Другое» поле скрывается, но введённый текст **остаётся**
  в состоянии (`st.other`) — при повторном выборе снова показывается.

**Что уходит на сервер (`buildAnswers()`):**

- Поле `other` добавляется в запись ответа, только если «Другое» **выбрано**
  И текст после `trim()` непустой.
- Если «Другое» выбрано, но текст пуст — отправляется только `id: "other"`,
  без `other`.

```jsonc
// single с заполненным «Другое»
"q_role":   { "value": "other", "other": "Технический директор" }
// multi с «Другое» среди прочих
"q_tools":  { "values": ["cursor", "other"], "other": "Aider" }
```

## 6. Финальная страница

**Экран `#thanks`** (центрировано, текст по центру):

- Бейдж-галочка в кружке (анимация `pop`: scale 0.5 → 1, 0.4s), цвет — `success`.
- Заголовок **«Спасибо»** (шрифт Unbounded).
- Тело: **«Ваш ответ записан.»**
- Подпись (моноширинный): **«Результаты войдут в открытое исследование.»**

**Вариант «уже проходил».** Если при загрузке найден флаг `vibesurvey_done`
(`localStorage` или cookie), интро/опрос пропускаются и сразу показывается
`#thanks`, но с изменёнными текстами:

- Заголовок → **«Вы уже проходили опрос»**
- Тело → **«Спасибо — ваш ответ уже записан.»**
- Подпись → пусто.

**Состояние ошибки отправки** (экран не меняется): кнопка «Отправить» выходит из
состояния загрузки, снизу всплывает тост (`is-error`, авто-скрытие 4.2с):
«Не удалось отправить. Проверьте соединение и попробуйте ещё раз.» Пользователь
может повторить отправку.

## 7. Дизайн: десктоп и мобильный

Светлая минималистичная тема, без тёмной. Лёгкая SVG-зернистость на фоне
(`feTurbulence`, opacity ~0.035). Уважается `prefers-reduced-motion`.

|---|---|---|
| `--page` | `#FBFBF9` | фон страницы (тёплый off-white) |
| `--card` | `#FFFFFF` | карточки/опции |
| `--ink` | `#15161A` | основной текст |
| `--muted` | `#6B6F76` | вторичный текст |
| `--faint` | `#9A9DA3` | третичный текст/хинты |

| `--accent-ink` | `#4A45C9` | акцент-hover/текст |

**Типографика (Google Fonts):**

| Роль | Шрифт | Где |
|---|---|---|
| `--font-display` | **Unbounded** (500–700) | вордмарк, заголовки, крупные номера |
| `--font-ui` | **Inter** (400–600) | тело, заголовки вопросов, опции |
| `--font-mono` | **JetBrains Mono** (400–500) | мета, кикер, счётчики, клавиши |

**Прочее:** радиусы — карточка 18, опция 13, кнопка 12, пилюля 999;
макс. ширина контента `--maxw: 640px`; ширина рейла `--rail-w: 38%`;
ease `cubic-bezier(.4, 0, .2, 1)`.

### 7.2. Адаптив (брейкпоинты)

| Ширина | Раскладка |
|---|---|
| **≥ 1280px** | Десктоп: 2 колонки (рейл 38% + сцена). Хинты клавиш видны. 2-колоночные опции. |
| **1024–1280px** | То же, рейл сужается до 34% (`--rail-w: 34%`). |
| **≤ 1023px** | Мобайл/планшет: 1 колонка, рейл **скрыт**. Появляется sticky-хедер (`#mhead`) с блюром. Нав-панель **прилипает к низу** (fixed, блюр, `safe-area-inset`). 2-колоночные опции → **1 колонка**. Хинт «Enter далее» скрыт. На интро футер-кнопка на всю ширину. |

### 7.3. Ключевые компоненты

- **Опция (`.opt`):** карточка-кнопка, слева индикатор-«марка» (круг для radio /
  скруглённый квадрат для check), по центру лейбл, справа хинт клавиши
  (`1`–`9`). Выбранная: фон `accent-soft`, граница `accent`, марка заливается
  акцентом с белой галочкой/точкой. Hover — граница `faint`, лёгкий `scale`
  при нажатии.
- **Поле «Другое»:** анимация раскрытия через `grid-template-rows` (см. §5),
  граница `accent-line`, счётчик «X/80».
- **Кнопки:** primary (акцент, белый текст, стрелка едет вправо на hover) и ghost
  (для «Назад»). У primary при отправке — спиннер и текст «Отправляем».
- **Топ-полоса прогресса:** фиксированная, 3px, появляется только в опросе.
- **Левый рейл:** sticky на всю высоту, мягкий вертикальный градиент к
  `accent-soft`, крупный номер вопроса шрифтом Unbounded.
- **Тосты:** по центру снизу, тёмные (или красные при ошибке), авто-скрытие 4.2с,
  кнопка закрытия.
- **Анимации:** `qin` (появление вопроса), `pop` (бейдж финала), `spin`
  (загрузка кнопки), `toastin` (тост). Все глушатся при reduced-motion.
- **Доступность:** `role="radiogroup"/"group"`, `role="radio"/"checkbox"`,
  `aria-checked`, единый focus-visible-ринг, `aria-live` для тостов.

## 8. Бэкенд-контракт и анти-спам

Прототип отдаётся **nginx**; статика + два API-эндпоинта.

### `GET /api/session`

Выдаёт одноразовый nonce и версию схемы. Ответ:

```json
{ "nonce": "<base64 от {ts, sig}>", "schema_version": 1 }
```

`nonce` — base64-обёртка над `{"ts": <unix>, "sig": "<hmac>"}`: подписанная
временная метка (анти-replay / анти-CSRF). Клиент получает её в `boot()` и
прикладывает к отправке.

### `POST /api/submit`

Принимает только POST (иначе `405`). Тело:

```json
{
  "schema_version": 1,
  "nonce": "<из /api/session>",
  "hp": "",
  "answers": {
    "q_role":  { "value": "developer" },
    "q_users": { "values": ["clients", "colleagues"] },
    "q_tools": { "values": ["cursor", "other"], "other": "Aider" }
  }
}
```

Правила формирования `answers` (`buildAnswers()`):

- `single` → `{ "value": "<id>" }`; `multi` → `{ "values": ["<id>", …] }`.
- Непустое «Другое» добавляет `"other": "<текст>"` (см. §5).
- Необязательные вопросы без выбора **не включаются** в payload.

Успех — `{ "ok": true }` (HTTP 200) → клиент ставит `vibesurvey_done` и
показывает «Спасибо». Иначе — тост об ошибке, можно повторить.

### Анти-спам (три рубежа)

1. **Honeypot** `#hp` — скрытое поле; бот его заполнит → сервер отклонит.
2. **Подписанный nonce** из `/api/session` — против replay/CSRF; сервер обязан
   валидировать подпись и срок.
3. **Дедупликация** — флаг `vibesurvey_done` (localStorage + cookie 1 год)
   против повторного прохождения с того же устройства.

> На бэкенде платформы нужно реализовать оба эндпоинта (выдача/валидация nonce и
> приём ответов с проверкой honeypot, подписи, версии схемы) и хранилище ответов
> (агрегируемое, обезличенное).

[`bitrix24/templates-dashboard`](https://github.com/bitrix24/templates-dashboard)
— «kickstart»-шаблон на **Nuxt 4** с библиотекой **Bitrix24 UI**. Даёт
SSR/Nitro-сервер, готовую тему, мультистраничность, командную палитру, светлую/

**Стек (из `package.json`, точные версии):**

| Bitrix24 UI/SDK | `@bitrix24/b24ui-nuxt ^2.6`, `@bitrix24/b24icons-vue ^2.0`, `@bitrix24/b24jssdk ^1.0` + `@bitrix24/b24jssdk-nuxt` |
| Данные/валидация | `zod ^4`, `@tanstack/vue-table ^8`, `@vueuse/core ^14` |

### Рекомендуемое отображение опроса на платформу

Опрос — **публичный и анонимный**, поэтому встроенная Bitrix24-часть (scopes,
iframe-install) нужна не везде. Делим на два контура:

**A. Публичный сбор ответов (без авторизации):**

```
app/
  pages/
    index.vue            // экраны Интро / Опрос / Спасибо (перенос UX из §3, §7)
  components/survey/
    OptionButton.vue     // .opt (radio/check, хинт клавиши)
    OtherField.vue       // раскрывающееся поле «Другое» (§5)
    SurveyRail.vue       // левый рейл (десктоп)
    MobileHeader.vue     // sticky-хедер (мобайл)
  composables/
    useSurvey.ts         // состояние answers/current, навигация, валидация, persist
server/
  api/
    session.get.ts       // выдача подписанного nonce + schema_version
    submit.post.ts       // приём ответов: honeypot + nonce + zod-валидация → хранилище
  utils/nonce.ts         // HMAC-подпись/проверка
public/
  survey-schema.json     // контент опроса (из docs/reference)
i18n/                    // ru (+ при необходимости др. языки)
```

- Тексты вопросов — из `survey-schema.json`; **zod**-схема валидирует входящие
  ответы по `schema_version`.
- Дизайн-токены из §7 кладём в Tailwind-тему/CSS-переменные; компоненты `b24ui`
  используем там, где совпадают по виду, иначе — кастомные (опции/рейл
  специфичны и проще свои).
- `localStorage`-логику (`progress`, `done`) и клавиатуру переносим 1:1.

**B. (Опционально, по желанию) Встроенный дашборд результатов в Bitrix24:**
именно здесь templates-dashboard раскрывается — сайдбар, командная палитра,
`@tanstack/vue-table` и графики `@unovis` для агрегированных результатов
исследования. Этот контур уже использует scopes `crm`/`user_brief` и
install-flow шаблона.

> Если встроенный дашборд не нужен — берём из шаблона только Nuxt-каркас,
> Nitro-`server/api`, тему и тулинг (ESLint/TS/i18n), а Bitrix24-iframe-часть не
> подключаем.

## 10. Деплой на сервер — по образцу templates-mcp

[`bitrix24/templates-mcp`](https://github.com/bitrix24/templates-mcp) даёт
production-паттерн деплоя Nuxt-приложения в Docker за reverse-proxy с TLS — его и
берём за образец (сам MCP-функционал нам не нужен, нужен **процесс деплоя**).

**Требования рантайма:** Node.js **≥ 22**, **pnpm v11** (через
`corepack enable`), Docker для прода.

**Сборка и запуск:**

```bash
corepack enable && pnpm install
pnpm build
pnpm start          # Nitro-сервер слушает :3000 (API + статика)
```

**Docker / reverse-proxy (паттерн из templates-mcp):**

- `docker-compose.yml` — **nginx-proxy + acme-companion**: автоматический
  Let's Encrypt TLS. Рекомендованный прод-вариант.
- `docker-compose.example.yml` — одиночный хост со «своим» TLS (когда reverse-
  proxy/сертификаты уже есть).
- Гайды по reverse-proxy (nginx-proxy / Caddy / nginx+certbot / Traefik) — в
  `docs/REVERSE-PROXY.md` шаблона.

**Переменные окружения (`.env`, из `.env.example`):** у нашего опроса состав
другой, но принцип тот же — секреты только через env, никогда в код. Минимум:

| Переменная | Назначение |
|---|---|
| `NUXT_SURVEY_SIGNING_SECRET` | секрет для HMAC-подписи nonce (`openssl rand -hex 32`) |
| `NUXT_SURVEY_STORAGE_URL` | подключение к хранилищу ответов (БД) |
| `NUXT_LOG_LEVEL` | уровень логирования (`info`/`warning`/`error`) |

**Health-check:** публичный эндпоинт вроде `GET /api/health` (как в templates-mcp)
для проверки готовности контейнера:

```bash
curl https://<домен>/api/health
```

**Шаги выкладки:**

1. Заполнить `.env` (секрет подписи, хранилище).
2. Поднять reverse-proxy с TLS (nginx-proxy+acme или свой).
3. `docker compose up -d`.
4. Проверить `/api/health` и прохождение опроса от интро до «Спасибо».

## 11. Требования к проекту и критерии приёмки

**Качество кода (по образцу обоих шаблонов):**

- `pnpm lint` (ESLint) и `pnpm typecheck` — без ошибок.
- Unit-тесты (Vitest) на ключевую логику: `buildAnswers()`, правила
  `exclusive`/`other`, валидация обязательных, формирование payload.
- CI (GitHub Actions): `lint` + `test` (+ опционально `build`) на PR.
- Секреты — только через env; honeypot, подпись nonce и версия схемы
  валидируются на сервере.

**Функциональные критерии приёмки:**

- [ ] Интро → 25 вопросов → «Спасибо» проходится мышью и клавиатурой.
- [ ] Прогресс (топ-полоса, рейл, мобильный хедер, счётчики) корректен на каждом шаге.
- [ ] `single`/`multi`, `exclusive`, `columns:2`, `required:false` работают по §4.
- [ ] «Другое»: раскрытие, авто-фокус, счётчик X/80, отправка текста по §5.
- [ ] Валидация блокирует «Далее» на пустом обязательном с подсветкой хелпера.
- [ ] Прогресс сохраняется и восстанавливается из `localStorage`.
- [ ] Повторное прохождение заблокировано (флаг `done`), показывается спец-экран.
- [ ] Дизайн совпадает с §7 на десктопе (≥1280 / 1024–1280) и мобайле (≤1023).
- [ ] `/api/session` выдаёт подписанный nonce; `/api/submit` валидирует и сохраняет.
- [ ] Деплой в Docker за TLS-прокси, `/api/health` отвечает.
