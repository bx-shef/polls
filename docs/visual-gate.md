# Визуальный гейт (issue #13)

Детерминированная визуальная верификация UI контура A: **render → screenshot →
сверка с эталоном**. Расхождение пикселей выше порога = провал. Гейт превращает
правило «UI не готов, пока не увиден глазами» из ритуала в автоматический барьер.

> Статус: **инфраструктура поднята** на фикстуре-заглушке (Nuxt/b24ui-слоя ещё нет).
> Машинерия доказана end-to-end — чтобы реальные экраны сразу делались под гейтом.
> Когда появятся экраны, фикстуры заменятся на маршруты приложения (см. ниже).

## Из чего состоит

| Часть | Файл | Роль |
|---|---|---|
| Конфиг | `playwright.config.ts` | проекты-брейкпоинты (desktop/mobile), детерминизм, путь эталонов |
| Реестр поверхностей | `test/visual/screens.visual.ts` | список проверяемых экранов + `toHaveScreenshot` |
| Фикстуры | `test/visual/fixtures/*.html` | временные заглушки экранов (уйдут с приходом Nuxt) |
| Эталоны | `test/visual/__screenshots__/**/*.png` | закоммичены, сверяются в каждом прогоне |
| Stop-хук | `.claude/hooks/visual-gate.sh` | прогоняет гейт на изменениях UI, блокирует Stop при провале |

## Команды

```bash
pnpm visual:install        # поставить chromium (в удалённой среде уже предустановлен).
                           # В «чистой» среде без системных .so: npx playwright install --with-deps chromium
pnpm test:visual           # прогон скриншот-регрессии (сверка с эталоном)
pnpm test:visual:update    # ОСОЗНАННО обновить эталоны. ОБЯЗАТЕЛЬНО затем:
                           # git diff test/visual/__screenshots__ — сверить .png глазами перед коммитом
```

Эталоны лежат по `snapshotPathTemplate` из `playwright.config.ts`, напр.:
`test/visual/__screenshots__/screens.visual.ts/intro-desktop.png` (и `-tablet`, `-mobile`).

## Как работает Stop-хук

Зарегистрирован в `.claude/settings.json` (событие `Stop`). На завершении сессии:

1. **Узкий триггер.** Смотрит, тронуты ли UI-поверхности (`app/ pages/ components/
   layouts/ composables/ assets/ public/ test/visual/`, `playwright.config.*`). Чистое
   ядро (`src/`, `docs/`, миграции) — мгновенный выход, гейт не запускается.
2. **Мягкая деградация.** Нет браузера (локальный dev без `pnpm visual:install`) —
   пропуск без блокировки (инфраструктурная нехватка ≠ регрессия). В удалённой среде
   chromium предустановлен (`PLAYWRIGHT_BROWSERS_PATH`).
3. **Блок только на реальном провале.** `pnpm test:visual` красный → `exit 2`, причина
   возвращается агенту; остановка заблокирована, пока UI не сверён.

> **Безопасность.** Хук и скрипт `test:visual` берутся из рабочего дерева ТЕКУЩЕЙ ветки
> (та же модель доверия, что у `session-start.sh`). На недоверенной ветке ревьюйте
> изменения в `.claude/hooks/` и в `package.json`-скриптах перед запуском сессии.

## Детерминизм (почему скриншоты не «плывут»)

- `reducedMotion: 'reduce'` + `animations: 'disabled'` — без анимаций/переходов.
- Три брейкпоинта как отдельные проекты — отдельный эталон на каждый (`docs/design.md` §8):
  **desktop** 1280 (рейл ~38%), **tablet** 1024–1279 (2 колонки, рейл ~34%), **mobile**
  (Pixel 7 — chromium-совместимый mobile-профиль: `isMobile`/`hasTouch`/mobile-UA, чтобы
  `@media (hover:none)`/`(pointer:coarse)` рендерились как на устройстве).
- **Системные шрифты** в фикстурах (без веб-фонтов) — рендер не зависит от загрузки сети.
- `maxDiffPixelRatio: 0.002` — терпит субпиксельный антиалиасинг текста, ловит регрессию
  раскладки/цвета.

**Версия chromium — неявный контракт среды.** Эталоны генерируются и сверяются в одной
среде (предустановленный chromium). При смене ОС/версии браузера рендер «плывёт» молча
(гейт пока не в CI — некому заметить): после такого обновления нужно осознанно
перегенерировать (`pnpm test:visual:update` → сверить `.png` глазами → коммит). Тёмная
тема — будущая работа (отдельные проекты `colorScheme:'dark'`, когда у b24ui-экранов появится dark).

## Как добавить экран (с приходом Nuxt-слоя)

1. Добавить в `playwright.config.ts` `webServer` (поднимает dev-сервер приложения) и `baseURL`
   (сейчас их нет — фикстуры грузятся через `file://`).
2. В `screens.visual.ts` заменить `file://`-фикстуру на маршрут (`page.goto('/s/:key')`),
   добавить состояния: пусто / ошибка / загрузка, hover/focus/disabled, **тёмная тема**.
3. `pnpm test:visual:update` → глазами проверить новые `.png` → закоммитить эталоны.
   Первый прогон нового экрана БЕЗ эталона всегда падает («snapshot doesn't exist») — это
   ожидаемо: прочитайте ошибку и сгенерируйте эталон через `:update`.
4. Фикстуры-заглушки из `test/visual/fixtures/` удалить, когда их экран реализован (#34).

## Приватность

Скриншоты — только dev/staging с **мок-данными**. Рендер живого портала (CRM/домены/
токены) не коммитим и не отправляем в облачный чат (см. `CLAUDE.md` → «Визуальная
верификация UI»).

## CI

Сейчас гейт — агент-сайд (Stop-хук) в предустановленной среде. Подключение к CI
(`pnpm check`) — когда появятся экраны и закреплённый контейнер рендера: иначе разница
шрифтов/версий браузера между средами даёт ложные провалы. До тех пор `pnpm check`
(typecheck + vitest + verify) визуальные тесты не трогает.
