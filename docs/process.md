# Как работает приложение — простыми словами

Документ для быстрого понимания: что происходит «от и до», без погружения в код.
Технические детали — в [брифе](./brief.md) и [модели данных](./data-model.md).

## Два контура

Сервис состоит из двух экранов-«контуров»:

- **Контур A — прохождение опроса** (`/s/:ключ`): то, что видит клиент/сотрудник.
  Интро → вопросы → «Спасибо».
- **Контур B — дашборд аналитики** (`/d/:ключ`): то, что видит сотрудник компании.
  NPS / CSAT / распределения / тренды / срезы по услуге·клиенту·направлению·ответственному.

## Жизненный цикл — пошагово

### 1. Опрос создаётся и «публикуется»
Анкета (вопросы, блоки, тексты экранов) компилируется в **иммутабельную версию** с
номером. Опубликованную версию нельзя переписать — можно только выпустить новую (v2, v3…).
У каждого вопроса стабильный «ключ» — благодаря ему ответы на v1 и v2 **сопоставимы**
между собой (аналитика не ломается при изменении анкеты).

### 2. Клиент получает ссылку и проходит опрос (контур A)
- При открытии страница запрашивает у сервера **текущую версию** анкеты для рендера
  (`GET /api/survey/:key/current`) — наружу отдаётся публичная проекция, без внутренней
  CRM-конфигурации.
- Перед стартом страница берёт у сервера одноразовый **nonce** (`GET /api/session`) —
  анти-повтор.
- Клиент отвечает: одиночный/множественный выбор, «Другое — свой вариант», текст.
  Прогресс сохраняется в браузере (можно перезагрузить и продолжить; deep-link `?q=N`).
- На «Отправить» уходит `POST /api/submit`. Сервер прогоняет конвейер защиты:
  honeypot → rate-limit → проверка nonce (анти-replay) → проверка версии → **валидация
  ответов** (устойчива к подделке ключей) → приглашение (если есть токен — одноразовый
  снимок CRM-контекста) → **запись** с серверными `id` и временем отправки.
- Контекст ответа (какая сделка/клиент/услуга) берётся из **приглашения**, а не от
  клиента — клиент не может его подменить.

### 3. Ответы агрегируются (контур B)
Дашборд считает метрики **на лету** поверх сохранённых ответов:
- **NPS / CSAT / CES**, распределения ответов (с человекочитаемыми метками опций).
- **4 уровня**: по опросу / услуге / клиенту / направлению + KPI ответственных.
- **Тренд NPS** помесячно (корректно через границу версий).
- **Срезы** по измерениям (услуга, направление, ответственный, клиент).
- **Фильтр по версии** (`?version=N`) — сравнение «до/после публикации».

### 4. Защита приватности — встроена
- **Малые выборки подавляются** (порог анонимности): если по срезу/месяцу слишком мало
  ответов, цифры не показываются — нельзя «вычислить» конкретного человека.
- **Tenant-изоляция**: данные каждого портала Bitrix24 (`portalId`) строго отделены.
- **Дашборд закрыт авторизацией**: в проде нужна подписанная сессия портала, иначе
  доступ закрыт (fail-closed) — имена клиентов/сотрудников не утекут.

## Связка с Bitrix24 (целевой сценарий)

1. **Триггер**: сделка переходит в нужную стадию → Bitrix24 шлёт вебхук → создаётся
   приглашение со снимком CRM-контекста (клиент, услуга, ответственный).
2. **Опрос**: клиент получает ссылку, проходит контур A.
3. **Результат → CRM**: ответ ложится в аналитику; KPI по ответственным/направлениям
   виден на дашборде; (в перспективе) результат пишется в таймлайн сделки и Живую ленту.
4. **Дашборд в портале**: открывается во фрейме Bitrix24 — портал передаёт авторизацию
   (handshake), сервис проверяет её и выдаёт сессию (cookie), привязанную к этому порталу.

## Что уже работает, а что — на этапе деплоя

- ✅ **Работает и проверяемо сейчас**: оба контура (опрос + дашборд) на встроенных
  демо-данных — можно запустить и показать (`pnpm dev` или docker, см.

  открытые задачи — [issues.md](./issues.md).
