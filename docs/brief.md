## 1. Что это и зачем

**Продукт.** Анонимный одностраничный опрос-исследование. Движок **не привязан к
теме**: весь контент (тексты, варианты, блоки) задаётся декларативно в
JSON-схеме, поэтому платформа переиспользуема под любой опрос. Тема и
формулировки референса при переносе обезличиваются (решение №4).

**Ключевые свойства:**

| Свойство | Значение |
|---|---|
| Анонимность | Без персональных данных, без авторизации |
| Длительность | ~5–7 минут |
| Объём | 25 вопросов, 8 тематических блоков |
| Типы вопросов | Один (`single`), несколько (`multi`), свободный текст (`text`) |
| Спец-механики | «Другое + свой текст», взаимоисключающие варианты, необязательные вопросы |
| Сохранение прогресса | `localStorage`, можно вернуться и дозаполнить |
| Защита от повтора | После отправки опрос повторно недоступен |
| Хранилище | Отдельная БД — **PostgreSQL** (решение №2) |
| Аналитика | Закрытый **дашборд результатов** внутри Bitrix24 (решение №1) |
| Язык | Один опрос = один язык; платформа поддерживает разные языки (решение №3) |
| Результат | Агрегированные **обезличенные** данные (решение №4) |

**Цель брифа** — воспроизвести UX и **структуру** опроса (контент при этом
обезличивается и задаётся заново) на стеке `templates-dashboard`
(Nuxt 4 + Bitrix24 UI), собрать к нему **закрытый дашборд результатов** внутри
Bitrix24, хранить ответы в **PostgreSQL** и развернуть всё на домене
**`polls.bx-shef.by`** по производственному паттерну `templates-mcp`
(Docker + reverse-proxy + TLS).

> 🎨 Визуальная проработка экранов и блоков на компонентах **Bitrix24 UI (b24ui)**
> с примерами кода — в отдельном документе [`docs/design.md`](./design.md).

## 2. Архитектура референс-прототипа

Прототип — **статический SPA без сборки**, отдаётся nginx. Четыре артефакта:

```
index.html            // каркас 3 экранов: #intro / #survey / #thanks
survey.css            // дизайн-токены и компоненты (~665 строк)
survey-app.js         // вся логика, vanilla JS, IIFE (~432 строки)
survey-schema.json    // контент: интро, блоки, 25 вопросов, тексты финала
```

Плюс два серверных эндпоинта (см. §8): `GET /api/session`, `POST /api/submit`.

**Принцип:** схема грузится через `fetch('./survey-schema.json')`, затем
`boot()` строит состояние и рендерит вопросы по одному. Контент полностью
отделён от логики — менять опрос можно без правки кода.

Структура схемы зафиксирована в обезличенном шаблоне
[`docs/reference/survey-schema.template.json`](./reference/survey-schema.template.json):
сохранены типы вопросов, спец-флаги, число вариантов и разбивка по блокам, а
оригинальные формулировки заменены нейтральными плейсхолдерами — **контент
задаётся заново при настройке опроса** (решение №4, §12).

**Состояние (в памяти + `localStorage`):**

```js
answers = [ { picked: [/* индексы выбранных опций */], other: "" }, … ] // по одному на вопрос
current = 0                                                              // индекс текущего вопроса
```

- `localStorage["survey_progress"]` — снапшот `answers` (пишется на каждое
  изменение) → возобновление сессии.
- `localStorage["survey_done"]` + cookie `survey_done=1` (срок 1 год) →
  блокировка повторного прохождения.

## 3. Пользовательский поток и протокол стадий

Три экрана-секции переключаются классом `.is-active`; на теле `#app` —
модификатор `.survey-on` (показывает верхнюю полосу прогресса).

```
ИНТРО ──[Начать опрос]──▶ ОПРОС (вопрос 1 … 25) ──[Отправить]──▶ СПАСИБО
                              ▲              │
                              └─ Назад ◀─────┘
```

### 3.1. Экран ИНТРО (`#intro`)

Что показано (все тексты — из блока `intro` схемы):
- Вордмарк (бренд) с акцентной точкой + метка года (моно).
- Кикер — надзаголовок (моно).
- Крупный заголовок (Unbounded, в несколько строк, `white-space: pre-line`).
- Лид-абзац с описанием.
- Ряд «чипов» из `intro.meta` (напр. «Анонимно», «~N минут», «Без перс. данных»).
- CTA-кнопка (`intro.cta`) + подпись с числом вопросов/блоков.
- Скрытый honeypot-input `#hp` (анти-бот, см. §8).

Переход по клику CTA → `startSurvey()`: с `#intro` снимается `is-active`,
`#survey` получает `is-active`, `#app` получает `survey-on` (появляется
топ-полоса), `current = 0`, рендер первого вопроса.

### 3.2. Экран ОПРОС (`#survey`)

**Постоянный «хром» (не зависит от вопроса):**

- **Топ-полоса прогресса** (`#topfill`) — ширина `= (current+1)/25 × 100%`.
- **Левый рейл (только десктоп, 38% ширины):** бренд, имя блока (`#railBlock`),
  крупный номер `#railNum` «01 / 25», прогресс-бар (`#railFill`), счётчик
  `#railCount` «Вопрос N из 25», подсказки клавиш внизу.
- **Мобильный хедер (`#mhead`, только мобайл, sticky):** имя блока (`#mBlock`) +
  номер `#mNum` «01 / 25».

**Контент вопроса (рендерится в `#stage` на каждый шаг):**

1. `q__meta` — «01 / 25 — ИМЯ БЛОКА» (на мобайле скрыт, дублируется в хедере).
2. `q__title` — текст вопроса.
3. `q__helper` — подсказка: **«Один вариант»** (single) / **«Можно выбрать
   несколько»** (multi) с иконкой; для необязательных — тег **«необязательно»**.
4. `opts` — кнопки-варианты (radio/checkbox), опционально в 2 колонки; у первых
   девяти — хинт клавиши `1`–`9`; у «Другое» — раскрывающееся текстовое поле.
5. `nav` — **Назад** (выключена на Q1) · хинт «Enter далее» · **Далее**
   (на последнем вопросе — **Отправить**).

**Что обновляется при каждом переходе (`updateChrome()` + `render()`):**

| Элемент | Обновление |
|---|---|
| `#topfill`, `#railFill` | ширина прогресса в % |
| `#railNum` | «NN / 25» |
| `#railBlock`, `#mBlock` | имя текущего блока |
| `#railCount` | «Вопрос N из 25» |
| `#mNum` | «NN / 25» |
| `#stage` | новый вопрос с анимацией `qin` (fade + сдвиг 6px, 0.2s) |
| прокрутка | сбрасывается в начало (`scrollTop`) |

**Границы блоков.** Имя блока (`railBlock`/`mBlock`/`q__meta`) меняется при
пересечении границ блоков — соответствие «вопрос → блок» см. §4.

### 3.3. Экран СПАСИБО (`#thanks`)

После успешной отправки: с `#survey` снимается `is-active`, с `#app` —
`survey-on` (топ-полоса скрывается), `#thanks` получает `is-active`. См. §6.

### 3.4. Навигация и клавиатура

- **Кнопки:** Назад / Далее(Отправить). На Q1 «Назад» выключена.
- **Клавиатура (десктоп):** `1`–`9` — выбрать вариант; `Enter` или `→` — далее;
  `←` — назад. Блокируется, пока фокус в текстовом поле «Другое».
- **Валидация при «Далее»:** если вопрос обязателен и ничего не выбрано —
  переход блокируется, `q__helper` краснеет (`is-error`), текст меняется на
  «Выберите вариант» (single) / «Выберите хотя бы один вариант» (multi).
  Необязательные (блок 8) пропускаются без выбора.
- **Deep-link для превью:** `?q=N` или `#qN` (N = 1…25) открывает опрос сразу
  на нужном вопросе — удобно для ревью макетов.

## 4. Структура вопросов (полная матрица)

**25 вопросов, 8 блоков.** Каждый вопрос: `key`, `block` (название), `type`
(`single`/`multi`/`text`), `metric`, `required`, `text`, массив `options`
(`key` + `label`, плюс флаги `isOther` / `isExclusive`), опционально `columns: 2`.
Поля `intro`/`thanks`/`blocks` шаблона — контракт фронта (рендер SPA); движок
(`SurveyDraft`) их игнорирует (zod strip).

> Контент обезличен (решение №4): ниже — **структура и назначение**, без
> оригинальных формулировок. Нейтральные имена блоков и ключи `q01…q25` совпадают с
> [шаблоном схемы](./reference/survey-schema.template.json).

- **`single` (18 вопросов):** один вариант, индикатор-«радио» (круг с точкой).
  Выбор заменяет предыдущий.
- **`multi` (7 вопросов):** несколько вариантов, индикатор-«чекбокс» (галочка):
  `q07`, `q11`, `q13`, `q14`, `q17`, `q18`, `q19`.
- **`columns: 2` (5 вопросов):** двухколоночная сетка вариантов на десктопе
  (`q11`, `q13`, `q14`, `q18`, `q19`). На мобайле всегда 1 колонка.
- **`exclusive` (3 опции):** по одному варианту-исключению в `q07`, `q11`, `q17`
  (тип «ничего/нет/не изменилось»). При его выборе снимаются все прочие; при
  выборе обычного — снимается исключающий.
- **`other` (7 вопросов):** вариант «Другое» со своим текстовым полем — см. §5.
- **`required: false` (4 вопроса, блок «Демография»):** можно пропустить,
  «Далее» не блокируется, пустые ответы на сервер **не отправляются**.

> Структура (типы, флаги, число вариантов, разбивка по блокам) — это
> спецификация; **формулировки задаются заново и обезличиваются**. Готовый
> нейтральный скелет: [`survey-schema.template.json`](./reference/survey-schema.template.json).
> Воплощение блоков на компонентах b24ui — в [`docs/design.md`](./design.md).

## 5. «Другое — свой вариант» (отдельный разбор)

Опция-«Другое» помечена в схеме флагом `"isOther": true` (всегда `key: "other"`,
`label: "Другое"`). Встречается в 7 вопросах: `q01`, `q11`, `q12`, `q13`,
`q14`, `q18`, `q19`.

**Разметка и поведение:**

1. Рендерится как обычная кнопка-вариант, но **сразу за ней** — скрытый
   раскрывающийся контейнер `.other-wrap[data-for=i]`.
2. При выборе «Другое» контейнеру добавляется `.is-open`; CSS анимирует
   `grid-template-rows: 0fr → 1fr` (0.25s), внутренник с `overflow: hidden` —
   поле плавно «выезжает».
3. Внутри `.other-field`: текстовый `input#otherinput`
   (`maxlength=80`, `placeholder="Свой вариант…"`) и счётчик символов
   `#othercount` формата **«X/80»**.
4. **Авто-фокус**: при выборе «Другое» поле получает фокус через 180 мс.
5. Значение хранится в `answers[current].other` и пишется в `localStorage` на
   каждое нажатие клавиши.

**Логика в зависимости от типа вопроса:**

- В `single`-вопросе выбор «Другое» (как любой вариант) снимает прочие — остаётся
  только «Другое» + текст.
- В `multi`-вопросе «Другое» сосуществует с другими (с учётом правил
  `exclusive`).
- При **снятии** выбора «Другое» поле скрывается, но введённый текст **остаётся**
  в состоянии (`st.other`) — при повторном выборе снова показывается.

**Что уходит на сервер (`buildAnswers()`):**

- Поле `text` добавляется в запись ответа, только если «Другое» **выбрано**
  И текст после `trim()` непустой.
- Если «Другое» выбрано, но текст пуст — отправляется только `values: ["other"]`,
  без `text`.

```jsonc
// single с заполненным «Другое» (single → values из одного элемента)
"q12": { "values": ["other"], "text": "<свой текст>" }
// multi с «Другое» среди прочих
"q18": { "values": ["q18_o4", "other"], "text": "<свой текст>" }
```

## 6. Финальная страница

**Экран `#thanks`** (центрировано, текст по центру):

- Бейдж-галочка в кружке (анимация `pop`: scale 0.5 → 1, 0.4s), цвет — `success`.
- Заголовок **«Спасибо»** (шрифт Unbounded).
- Тело: **«Ваш ответ записан.»**
- Подпись (моноширинный): **«Результаты войдут в открытое исследование.»**

**Вариант «уже проходил».** Если при загрузке найден флаг `survey_done`
(`localStorage` или cookie), интро/опрос пропускаются и сразу показывается
`#thanks`, но с изменёнными текстами:

- Заголовок → **«Вы уже проходили опрос»**
- Тело → **«Спасибо — ваш ответ уже записан.»**
- Подпись → пусто.

**Состояние ошибки отправки** (экран не меняется): кнопка «Отправить» выходит из
состояния загрузки, снизу всплывает тост (`is-error`, авто-скрытие 4.2с):
«Не удалось отправить. Проверьте соединение и попробуйте ещё раз.» Пользователь
может повторить отправку.

## 7. Дизайн: десктоп и мобильный

Светлая минималистичная тема, без тёмной. Лёгкая SVG-зернистость на фоне
(`feTurbulence`, opacity ~0.035). Уважается `prefers-reduced-motion`.

### 7.1. Дизайн-токены

**Палитра:**

| Токен | HEX | Назначение |
|---|---|---|
| `--page` | `#FBFBF9` | фон страницы (тёплый off-white) |
| `--card` | `#FFFFFF` | карточки/опции |
| `--ink` | `#15161A` | основной текст |
| `--muted` | `#6B6F76` | вторичный текст |
| `--faint` | `#9A9DA3` | третичный текст/хинты |
| `--line` / `--line-soft` | `#E7E7E2` / `#F0F0EC` | границы |
| `--accent` | `#5B5BD6` | акцент (индиго) |
| `--accent-ink` | `#4A45C9` | акцент-hover/текст |
| `--accent-soft` | `#EEEEFB` | фон выбранной опции |
| `--accent-line` | `#C9C8F2` | граница поля «Другое» |

## 8. Бэкенд-контракт и анти-спам

Прототип отдаётся **nginx**; статика + два API-эндпоинта.

### `GET /api/session`

Выдаёт одноразовый nonce и версию схемы. Ответ:

```json
{ "nonce": "<base64 от {ts, sig}>", "schema_version": 1 }
```

`nonce` — base64-обёртка над `{"ts": <unix>, "sig": "<hmac>"}`: подписанная
временная метка (анти-replay / анти-CSRF). Клиент получает её в `boot()` и
прикладывает к отправке.

### `POST /api/submit`

Принимает только POST (иначе `405`). Тело:

```json
{
  "schema_version": 1,
  "nonce": "<из /api/session>",
  "hp": "",
  "answers": {
    "q02": { "values": ["q02_o3"] },
    "q07": { "values": ["q07_o1", "q07_o2"] },
    "q18": { "values": ["q18_o4", "other"], "text": "<свой текст>" }
  }
}
```

Правила формирования `answers` (`buildAnswers()`):

- `single` → `{ "values": ["<id>"] }`; `multi` → `{ "values": ["<id>", …] }`
  (на сервере оба парсит `rawAnswerSchema`: одно поле `values: string[]`).
- Непустое «Другое» добавляет `"text": "<текст>"` (см. §5).

> Контракт реализован в ядре: `src/api/handlers.ts` (`createApi`). Payload расширен
> полями `surveyKey`/`versionNo` (мультиопросное ядро; пин версии — из приглашения
> или ссылки). `id`/`submittedAt` записи ставит сервер; клиентские значения игнорируются.
- Необязательные вопросы без выбора **не включаются** в payload.

Успех — `{ "ok": true }` (HTTP 200) → клиент ставит `survey_done` и
показывает «Спасибо». Иначе — тост об ошибке, можно повторить.

### Анти-спам (три рубежа)

1. **Honeypot** `#hp` — скрытое поле; бот его заполнит → сервер отклонит.
2. **Подписанный nonce** из `/api/session` — против replay/CSRF; сервер обязан
   валидировать подпись и срок.
3. **Дедупликация** — флаг `survey_done` (localStorage + cookie 1 год)
   против повторного прохождения с того же устройства.

> На бэкенде платформы реализуем оба эндпоинта (выдача/валидация nonce и приём
> ответов с проверкой honeypot, подписи и версии схемы) и пишем ответы в
> **PostgreSQL** (решение №2) — обезличенно, в виде, пригодном для агрегации в
> дашборде результатов (§9-B).

## 10. Деплой на сервер — по образцу templates-mcp

[`bitrix24/templates-mcp`](https://github.com/bitrix24/templates-mcp) даёт
production-паттерн деплоя Nuxt-приложения в Docker за reverse-proxy с TLS — его и
берём за образец (сам MCP-функционал нам не нужен, нужен **процесс деплоя**).

**Целевой контур:** домен **`polls.bx-shef.by`**, TLS Let's Encrypt через
nginx-proxy + acme-companion (решение №5), хранилище — **PostgreSQL** (решение №2).

**Требования рантайма:** Node.js **≥ 22**, **pnpm 10.33** (зафиксирован в
`package.json`, через `corepack enable`), Docker для прода.

**Сборка и запуск:**

```bash
corepack enable && pnpm install
pnpm build
pnpm start          # Nitro-сервер слушает :3000 (API + статика)
```

**Docker / reverse-proxy (паттерн из templates-mcp):**

- `docker-compose.yml` — **nginx-proxy + acme-companion**: автоматический
  Let's Encrypt TLS. Рекомендованный прод-вариант.
- `docker-compose.example.yml` — одиночный хост со «своим» TLS (когда reverse-
  proxy/сертификаты уже есть).
- Гайды по reverse-proxy (nginx-proxy / Caddy / nginx+certbot / Traefik) — в
  `docs/REVERSE-PROXY.md` шаблона.
- **PostgreSQL** добавляется сервисом в тот же `compose` (отдельный контейнер +
  том для данных); приложение ходит в неё по `NUXT_DATABASE_URL`.

> В этом репозитории `docker-compose.yml` — локальная БД для разработки;
> прод-compose (приложение + nginx-proxy + Postgres) и `docs/REVERSE-PROXY.md`
> живут в репозитории `bitrix24/templates-mcp` и собираются в фазе деплоя.

**Переменные окружения (`.env`, из `.env.example`):** у нашего опроса состав
другой, но принцип тот же — секреты только через env, никогда в код. Минимум:

| Переменная | Назначение |
|---|---|
| `NUXT_SURVEY_SIGNING_SECRET` | секрет для HMAC-подписи nonce (`openssl rand -hex 32`) |
| `NUXT_DATABASE_URL` | строка подключения к **PostgreSQL** (`postgres://…`) |
| `NUXT_B24_CLIENT_ID` / `NUXT_B24_CLIENT_SECRET` | Bitrix24-приложение (дашборд результатов) |
| `NUXT_LOG_LEVEL` | уровень логирования (`info`/`warning`/`error`) |

**Health-check:** публичный эндпоинт вроде `GET /api/health` (как в templates-mcp)
для проверки готовности контейнера:

```bash
curl https://polls.bx-shef.by/api/health
```

**Шаги выкладки:**

1. Заполнить `.env` (секрет подписи nonce, `NUXT_DATABASE_URL`, параметры B24).
2. Поднять reverse-proxy с TLS для `polls.bx-shef.by` (nginx-proxy + acme).
3. `docker compose up -d` (приложение + PostgreSQL).
4. Применить миграции БД.
5. Зарегистрировать локальное приложение Bitrix24 (Application/Installation URL,
   scopes `crm`, `user_brief`, `bizproc`, `log`, `im`, `user` — см. §13).
6. Проверить `/api/health`, прохождение опроса от интро до «Спасибо» и доступ к
   дашборду результатов.

## 13. Сценарии применения в Bitrix24

Движок опроса — это **«Feedback OS»**: один код, два режима развёртывания. Продаём
не анкеты, а **решения** — AI-слой превращает ответы (включая фритекст «Другое») в
число (CSAT/NPS), причину (тег) и следующее действие. Сам опрос — самый дешёвый
датчик; защищённый ров — это AI-анализ.

**Дорожная карта (решение №6):**
- **Клин №1 (платный) — Сценарий A:** опрос по результату сделки → AI → KPI.
- **Тариф захвата (бесплатный) — Сценарий B:** опросы сотрудников → Живая лента.
- Оба — на одном движке; различаются датчиком (клиент / сотрудник) и приёмником
  (таймлайн + KPI / Лента). **AI-анализ — общий мозг.**

### 13.1. Сценарий A — опрос по результату сделки → AI → KPI

```
Сделка → стадия «Успешна»
        │  (робот bizproc.robot.add, DOCUMENT_TYPE = DEAL)
        ▼
   наш HANDLER (server/api) ──► ссылка клиенту (Email / SMS / Открытые линии)
        │                              │
        │                              ▼
        │                       клиент отвечает (≤ 1–2 мин)
        ▼                              │
   AI-анализ verbatim ◄────────────────┘
   (CSAT/NPS + тема + тональность)
        │
        ├──► crm.timeline.logmessage.add → запись в сделку
        ├──► RETURN_PROPERTIES / bizproc.event.send → обратно в бизнес-процесс
        ├──► KPI ответственного → встроенный дашборд (§9-B)
        └──► петля 10x: детрактор → авто-задача recovery; промоутер → запрос отзыва
```

**Механика по шагам (методы сверены по REST-докам):**

1. **Установка робота** — `bizproc.robot.add` c
   `DOCUMENT_TYPE = ['crm','CCrmDocumentDeal','DEAL']`,
   `FILTER.INCLUDE = [['crm','CCrmDocumentDeal']]`, `USE_SUBSCRIPTION:'Y'`,
   `RETURN_PROPERTIES` (CSAT, тег-причина), `HANDLER` = наш `server/api/robot`.
   Руководитель ставит робота на стадию «Сделка успешна» **без кода**.
2. **Срабатывание** — сделка дошла до победы → робот шлёт на `HANDLER` ID сделки +
   `auth`. (Дешёвая альтернатива — событие `onCrmDealUpdate` с проверкой стадии.)
3. **Доставка клиенту** — короткая ссылка на опрос по выбранному каналу.
4. **Анализ** — AI считает CSAT/NPS, извлекает тему и тональность из verbatim.
5. **Возврат** — `crm.timeline.logmessage.add` (запись в сделку); значения обратно
   в бизнес-процесс через `RETURN_PROPERTIES` (`bizproc.event.send`); KPI
   ответственного — в дашборд.
6. **Замыкание петли (10x)** — детрактор → робот сам открывает задачу «recovery»
   руководителю + черновик извинения от AI; промоутер → авто-запрос отзыва/реферала.

**Риски и гарантии:**
- *Response rate.* Клиенты игнорируют анкеты → вход в **один тап** (NPS 0–10),
  выбор канала, правильный тайминг.
- *KPI ≠ кнут.* Иначе менеджеры начнут прятать недовольных. KPI —
  **анонимизированный**, статистически сглаженный, с фокусом на **восстановление
  клиента**, а не наказание.

### 13.2. Сценарий B — опросы сотрудников → Живая лента

**Механика:**
1. **Приглашение** — `im.notify.system.add` / `im.message.add` (ссылка на пульс-опрос).
2. **Сбор** — **анонимно** (eNPS, пульс-опросы, ретро).
3. **Публикация агрегата** — `log.blogpost.add`: `POST_TITLE` + `POST_MESSAGE`,
   адресаты `DEST` (`UA` — все, `DR` — подразделение, `SG` — рабочая группа),
   `IMPORTANT:'Y'` для пульс-итогов. Возвращает ID поста.

**Риск:** анонимность — это святое. В Ленту — **только агрегаты**, никогда
индивидуально, иначе данные умрут на втором опросе. Публикация — курируемо, не спам.
