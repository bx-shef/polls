# Процесс — как работает сервис опросов (простыми словами)

> Last reviewed: 2026-07-24
>
> Этот документ описывает **процесс**: как сервис работает от события в CRM до аналитики, и как
> оператору управлять опросами. Техническая карта (архитектура, данные, деплой, решения) — в
> [`project-map.md`](./project-map.md).

## Одна картинка

```
        Bitrix24 (CRM)                     polls                         Bitrix24 (портал)
   ┌────────────────────┐        ┌──────────────────────┐        ┌────────────────────────┐
   │ сделка → стадия     │  ───►  │ приглашение          │  ───►  │ клиент проходит опрос   │
   │ («Успешна» и т.п.)  │ триггер│ (снимок контекста     │ ссылка │ /s/:ключ  (контур A)    │
   └────────────────────┘        │  сделки: клиент/       │        └───────────┬────────────┘
                                  │  услуга/направление/   │                    │ ответы
                                  │  ответственный)        │                    ▼
                                  └──────────────────────┘        ┌────────────────────────┐
                                                                  │ аналитика /d/:ключ (B)  │
   Датчик (событие CRM)  ──────►  Опрос  ──────►  Аналитика ◄──────│ NPS/CSAT/срезы, KPI,    │
                                                                  │ (в перспективе Лента)   │
                                                                  └────────────────────────┘
```

**Формула:** Датчик (событие CRM) → Опрос → Аналитика (KPI / Лента портала). Опрос — самый дешёвый
датчик; ценность — в аналитике и AI-разборе ответов.

## Два контура

- **Контур A — прохождение опроса** (`/s/:ключ`). Кто: клиент или сотрудник. Что видит: экран Интро →
  вопросы → «Спасибо». Анонимно, без авторизации, ~5–7 минут, можно вернуться и дозаполнить.
- **Контур B — дашборд аналитики** (`/d/:ключ`). Кто: сотрудник компании (внутри Bitrix24). Что видит:
  NPS/CSAT/распределения ответов, тренд по месяцам, срезы по клиенту·услуге·направлению·ответственному,
  сравнение «до/после» между версиями анкеты. Закрыт авторизацией портала.

## Жизненный цикл — по шагам

### 1. Опрос создаётся и публикуется

Опрос собирается в конструкторе (админ-UI `/admin/surveys`) или программно и **компилируется** в
иммутабельную версию с номером. У вопросов и вариантов — стабильные ключи (`question_key`/`option_key`),
поэтому версии v1 и v2 **сопоставимы** в аналитике (правка текста — тот же ключ; смена смысла — новый
ключ). Опубликованную версию нельзя переписать.

### 2. Запуск опроса (датчик)

Опрос запускается по событию в CRM или вручную:

- **Автоматически по стадии сделки** (целевой сценарий, #17): сделка переходит в триггер-стадию
  (напр. `WON` / `C1:WON`) → Bitrix шлёт событие → сервер снимает контекст сделки (`crm.deal.get` +
  товары) → создаёт **приглашение** со снимком `CrmContext` (клиент/услуга/направление/ответственный) →
  отправляет ссылку клиенту.
- **Вручную из карточки** сделки или задачи (виджеты `CRM_DEAL_DETAIL_ACTIVITY` / `TASK_VIEW_SIDEBAR`):
  сотрудник жмёт «Запустить опрос» → тот же снимок контекста → приглашение + ссылка.

Приглашение **пинит версию** анкеты: публикация новой версии не влияет на уже отправленные приглашения.

### 3. Клиент проходит опрос (контур A)

Страница `/s/:ключ`:

1. берёт текущую опубликованную версию (`GET /api/survey/:key/current` — публичная проекция **без**
   внутренней CRM-конфигурации);
2. получает одноразовый `nonce` (`GET /api/session`);
3. клиент отвечает; прогресс сохраняется в браузере (`localStorage`), поддержан возврат и deep-link `?q=N`;
4. отправка `POST /api/submit` проходит конвейер защиты:
   **honeypot → rate-limit → nonce (анти-replay) → версия → валидация ответов → приглашение
   (токен → снимок CRM, одноразовый) → запись** с серверными `id`/временем.

Контекст берётся **из приглашения**, клиент его подменить не может. Повтор по тому же токену → `409`
(идемпотентность). После успешной отправки прогресс в браузере очищается, повтор недоступен.

### 4. Аналитика (контур B)

Дашборд `/d/:ключ` считает метрики на лету через ядро агрегации:

- **NPS** (% промоутеров − % детракторов), **CSAT** (среднее / доля топ-бокса), **CES**, распределения
  ответов (с человекочитаемыми метками, не ключами);
- **4 уровня**: по опросу · по услуге/товару · по клиенту в динамике (помесячно) · по направлению +
  KPI ответственных;
- **тренд NPS** помесячно — версионно-безопасно (через границу версий);
- **срезы** по измерениям (клиент/услуга/направление/ответственный) с денормализованными подписями;
- **фильтр по версии** (`?version=N`) — весь дашборд по одной версии для сравнения «до/после».

### 5. Приватность встроена

- **Малые выборки подавляются**: если по срезу меньше порога ответов (`ANONYMITY_THRESHOLD = 5`),
  метрика не показывается — защита от «NPS Иванова по одному ответу».
- **Tenant-изоляция**: каждый портал видит только свои данные (`portalId`).
- **Дашборд закрыт авторизацией** портала (fail-closed: без валидной сессии — не отдаём агрегаты).
- Опросы клиентов режем по сделке; опросы сотрудников (eNPS/пульс) — анонимно.

## Связка с Bitrix24 (кратко)

**Установка приложения** сохраняет OAuth-токены портала (шифрованы), регистрирует робота автоматизации
и виджеты. **Дашборд во фрейме** портала проходит handshake (`BX24.getAuth` → подписанная сессия-cookie,
привязанная к порталу). **Результат** возвращается в CRM: в перспективе — запись в таймлайн сделки (#18)
и агрегаты в Ленту портала (сценарий B). Технические детали связки и lifecycle портала (установка,
удаление, keep-alive токенов, защита от подделок) — в [`project-map.md`](./project-map.md) §Интеграция и
§Безопасность.

## Управление опросами (для оператора)

### Как устроен опрос

Опрос = черновик (`SurveyDraft`) → `compile()` → иммутабельная опубликованная версия. Черновик живёт в
редакторе-клиенте; «сохранить» = **опубликовать новую версию** (`currentVersionNo + 1`). Отдельной
таблицы черновиков нет.

### Три способа завести опрос

1. **Админ-UI** (`/admin/surveys`): список опросов (фильтр по типу сущности) + конструктор — правка
   заголовка/стадий-триггеров, добавить/удалить/переставить вопросы и опции, тип/метрика/обязательность/
   баллы → публикация. Под авторизацией (`requirePortalSession`, fail-closed).
2. **Сид** — `draftV*` в `src/demo/seed.ts` (демо-данные).
3. **Программно** — `await store.publish(draft, versionNo)`.

### Как добавить вопрос

1. В черновике добавить вопрос с **новым стабильным `question_key`** (латиница/цифры/`_`), выбрать тип
   (`single`/`multi`/`text`) и метрику (`nps`/`csat`/`ces`/`none`).
2. Опубликовать новую версию (`publish(updatedDraft, prevVersion + 1)`).
3. Старые ответы остаются на своих версиях; сопоставление — по `question_key`. Дашборд берёт вопросы
   **по метрике из текущей версии** (не по хардкод-ключам), так что новый вопрос сразу попадает в срезы.

### Какой опрос запускается по какой сущности

Настраивается через env (временный слой, пока нет UI-маппинга): `SURVEY_KEY_DEAL`, `SURVEY_KEY_TASK`,
`SURVEY_KEY_LEAD`, … с дефолтом `SURVEY_KEY_DEFAULT` (иначе `csat_postdeal`). Привязка к направлению —
через `invitationPolicy.triggerStages` (стадии-триггеры) в самой версии опроса.

### Как удалить данные за период

Ручная операция в БД (кнопка «очистить период» — отдельная задача), только по нужному порталу и периоду,
предварительно бэкап:

```sql
delete from response
 where portal_id = (select id from portal where member_id = '<member_id>')
   and submitted_at < '2026-01-01T00:00:00Z';
```

`response_product` удалится каскадом (FK); `survey_version` не трогаем. Полная очистка портала —
`delete from response where portal_id = …`.

### Дополнительные точки встройки

- **Реализовано:** авто-триггер `event.bind ONCRMDEALUPDATE` → `/api/b24/deal-update` (#17), плейсменты
  `CRM_DEAL_DETAIL_ACTIVITY` (карточка сделки), `CRM_ANALYTICS_MENU` (меню аналитики — сюда затянут дашборд),
  `TASK_VIEW_SIDEBAR` (карточка задачи).
- **Не реализовано:** робот `bizproc.robot.add` (параметры готовы, но эндпоинт `/api/b24/robot` нет —
  поэтому робот пока НЕ регистрируется при установке; триггер идёт через `event.bind`).
- **На будущее** (новые `placement.bind` поверх клиента, без переделки ядра): вкладка сделки
  `CRM_DEAL_DETAIL_TAB`, встройки лида/контакта/компании, левое меню, запись результата в таймлайн
  (`crm.activity.configurable.add`, #18), доставка ссылки через `imbot`.
